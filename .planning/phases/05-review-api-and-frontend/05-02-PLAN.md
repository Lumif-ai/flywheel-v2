1. **types/email.ts** — TypeScript interfaces matching the backend Pydantic models from Plan 01:
   - `Thread`: thread_id, subject, sender_name, sender_email, latest_received_at (string ISO), message_count, max_priority (number|null), priority_tier ('critical'|'high'|'medium'|'low'|'unscored'), has_pending_draft, draft_id (string|null), is_read
   - `ThreadListResponse`: threads (Thread[]), total, offset, limit
   - `Score`: priority, category, reasoning (string|null), suggested_action (string|null), context_refs (Array<{entry_id: string; file_name?: string; content_preview?: string}>)
   - `Message`: id, gmail_message_id, sender_email, sender_name, subject, snippet, received_at, is_read, is_replied, score (Score|null)
   - `Draft`: id, status ('pending'|'sent'|'dismissed'), draft_body (string|null), user_edits (string|null)
   - `ThreadDetailResponse`: thread_id, subject, messages (Message[]), draft (Draft|null), max_priority (number|null), priority_tier
   - `FlatItem`: discriminated union type `{type: 'header'; tier: string; label: string} | {type: 'thread'; thread: Thread}` for virtual list

2. **store/emailStore.ts** — Zustand store following the pattern in stores/ui.ts:
   - State: selectedThreadId (string|null), detailOpen (boolean), alertDismissedIds (Set<string>)
   - Actions: selectThread(id: string|null) — sets selectedThreadId and detailOpen=!!id; closeDetail() — sets both to null/false; dismissAlert(threadId: string) — adds to alertDismissedIds Set

3. **hooks/useEmailThreads.ts** — follow useBriefing.ts pattern exactly:
   - `useEmailThreads(params?: {priority_min?: number; offset?: number; limit?: number})` — useQuery with queryKey ['email-threads', params], queryFn calls api.get<ThreadListResponse>('/email/threads', {params}), staleTime 30_000

4. **hooks/useThreadDetail.ts**:
   - `useThreadDetail(threadId: string|null)` — useQuery with queryKey ['thread-detail', threadId], queryFn calls api.get<ThreadDetailResponse>(`/email/threads/${threadId}`), enabled: !!threadId, staleTime 30_000

5. **hooks/useDraftActions.ts** — three mutation hooks:
   - `useApproveDraft()` — useMutation calling api.post(`/email/drafts/${draftId}/approve`), onSuccess invalidates ['email-threads'] and ['thread-detail']
   - `useDismissDraft()` — useMutation calling api.post(`/email/drafts/${draftId}/dismiss`), same invalidation
   - `useEditDraft()` — useMutation calling api.put(`/email/drafts/${draftId}`, {draft_body}), same invalidation
   - Each returns the mutation hook directly. The mutationFn takes an object {draftId: string, ...} as appropriate.

6. **hooks/useManualSync.ts**:
   - `useManualSync()` — useMutation calling api.post('/email/sync'), onSuccess invalidates ['email-threads']

2. **ThreadList.tsx** — virtual scrolling container with priority tier group headers:
   - Props: threads (Thread[]), loading (boolean)
   - Internally compute `flatItems: FlatItem[]` by iterating sorted threads and inserting `{type: 'header', tier, label}` items at each priority tier boundary. Tier labels: "Critical Priority", "High Priority", "Medium Priority", "Low Priority", "Unscored".
   - Use `useVirtualizer` from @tanstack/react-virtual:
     - count: flatItems.length
     - getScrollElement: () => parentRef.current
     - estimateSize: (index) => flatItems[index].type === 'header' ? 40 : 80
     - overscan: 5
   - The scroll container div MUST have: ref={parentRef}, className="overflow-auto flex-1", style={{height: '100%'}}. This is CRITICAL — without a fixed/constrained height, virtualization does not work.
   - Each virtual row checks flatItems[index].type:
     - 'header': render a sticky-ish tier label (text-sm font-semibold text-muted-foreground px-4 py-2 with a light background)
     - 'thread': render <ThreadCard thread={...} isSelected={...} onSelect={...} />
   - Get selectedThreadId and selectThread from useEmailStore
   - Show loading skeleton if loading is true (simple pulsing rows)
   - Show empty state if threads.length === 0 and not loading

   - Props: draft (Draft)
   - Shows draft body (draft.user_edits ?? draft.draft_body) in a bordered text area (read-only by default, editable when edit mode toggled)
   - Three buttons: "Approve & Send" (primary, coral), "Edit" (secondary), "Dismiss" (ghost/destructive)

   - Edit mode: toggles textarea to editable, shows "Save" and "Cancel" buttons. Save calls editDraft mutation with the new text, then exits edit mode.
   - Approve calls approveDraft with draft.id. Show loading spinner on button during mutation.
   - Dismiss calls dismissDraft with draft.id.
   - After approve success, show a brief success indicator (change button text to "Sent!" for 2 seconds)
   - If draft.status !== 'pending', show status badge and disable all actions
