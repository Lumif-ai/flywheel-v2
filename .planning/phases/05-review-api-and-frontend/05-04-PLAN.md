Two anti-pattern fixes:

**Fix 1 — Dynamic tier colors in ThreadDetail.tsx (WARNING):**
In the MessageRow component, the priority badge (lines 88-89) hardcodes `TIER_COLORS.critical` for all messages regardless of actual score. Fix:

1. Add a helper function to map a numeric priority to a PriorityTier key:
```
function priorityToTier(priority: number): PriorityTier {
  if (priority >= 5) return 'critical'
  if (priority >= 4) return 'high'
  if (priority >= 3) return 'medium'
  return 'low'
}
```
This mirrors the backend tier mapping logic (5=critical, 4=high, 3=medium, 1-2=low).

2. In the score badge rendering section of MessageRow (inside `{message.score && (` block), replace:
```
color: TIER_COLORS.critical,
backgroundColor: `${TIER_COLORS.critical}1A`,
```
with:
```
const msgTier = priorityToTier(message.score.priority)
// then in style:
color: TIER_COLORS[msgTier],
backgroundColor: `${TIER_COLORS[msgTier]}1A`,
```
Compute `msgTier` once before the JSX return inside the score section, or use an inline expression.

**Fix 2 — Guard useEmailThreads in layout.tsx (WARNING):**
In AppShell (layout.tsx line 41), `useEmailThreads()` fires before the `isStandalone` check on line 43. On /onboarding and /invite routes, this triggers a GET /email/threads call that returns 401 (no auth context) — React Query retries it silently.

Fix: Move `useEmailThreads()` into a small wrapper component that only renders inside the authenticated shell. Approach:

1. Create a small inner component `AuthenticatedAlerts` that:
   - Calls `const { data: emailData } = useEmailThreads()`
   - Returns `emailData?.threads ? <CriticalEmailAlert threads={emailData.threads} /> : null`
2. In AppShell, remove the `useEmailThreads()` call from the top level
3. Remove the standalone `{emailData?.threads && <CriticalEmailAlert ... />}` JSX from both mobile and desktop branches
4. Instead render `<AuthenticatedAlerts />` inside each non-standalone branch (mobile and desktop), after `<AppRoutes />`

This ensures the hook only runs when the user is authenticated and on a non-standalone route. React Query deduplication still works since EmailPage's call uses the same query key.
