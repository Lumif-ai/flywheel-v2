       - Calls `self._request("post", "/api/v1/skills/telemetry", json={...})`
       - Wrap in try/except and return `{"status": "error"}` on failure -- telemetry must NEVER block skill execution. Log warning via `logger.warning()`.

    2. Modify `flywheel_run_skill` in `cli/flywheel_mcp/server.py` (lines 157-220):
       - After creating the `FlywheelClient` instance (line 185), add the cc_executable check BEFORE the `client.start_skill_run()` call
       - Logic:
         ```python
         # Check if skill is cc_executable -- redirect to in-context execution
         try:
             skills_resp = client.fetch_skills()
             skill_meta = next(
                 (s for s in skills_resp.get("items", []) if s["name"] == skill_name),
                 None,
             )
             if skill_meta and skill_meta.get("cc_executable", False):
                 # Log redirect telemetry (fire-and-forget, never blocks)
                 try:
                     client.log_skill_telemetry(skill_name, "redirect_to_in_context")
                 except Exception:
                     pass  # telemetry failure must not block redirect
                 return (
                     f"REDIRECT: '{skill_name}' runs better in-context using your Claude subscription.\n\n"
                     f"Instead of flywheel_run_skill, do this:\n"
                     f"1. Call flywheel_fetch_skill_prompt(skill_name='{skill_name}') to get the full prompt\n"
                     f"2. Execute the skill instructions directly in this conversation\n"
                     f"3. Save results via flywheel_save_document and flywheel_write_context\n\n"
                     f"This gives better results because you have full conversation context, "
                     f"user files, and codebase access."
                 )
         except Exception:
             pass  # On ANY failure to check, fall through to server-side execution (safe default per research pitfall 1)
         ```
       - IMPORTANT: The fallthrough on exception is deliberate. If fetch_skills fails, the tool proceeds with server-side execution as before. This is the safe default.
       - Do NOT modify any other part of flywheel_run_skill. The polling logic, error handling, and timeout remain identical.
       - Do NOT cache the skill list. Per-call fetch is <100ms and avoids stale cc_executable flags during migration.

    3. Also add telemetry logging for the SERVER-SIDE path. After a successful `client.start_skill_run()` returns a run_id (line 187), add:

       # Log server-side execution telemetry
