  <action>
    1. Add `flywheel_log_skill_execution` MCP tool to `cli/flywheel_mcp/server.py`:
       - Place it near the other skill-related tools (after `flywheel_run_skill`)
       - Tool signature:
         ```python
         @mcp.tool(output_schema=None)
         def flywheel_log_skill_execution(
             skill_name: str = "",
             status: str = "completed",
             context_files_written: str = "",
             documents_saved: str = "",
         ) -> str:
         ```
       - Docstring: "Log completion of an in-context skill execution for telemetry. Call this after executing a skill in-context (via flywheel_fetch_skill_prompt) to track adoption metrics. status: 'completed' or 'failed'. context_files_written: comma-separated list of context file names written to. documents_saved: comma-separated list of document titles saved."
       - Implementation:
         a. Create FlywheelClient instance
         b. Build metadata dict: `{"status": status, "context_files_written": [f.strip() for f in context_files_written.split(",") if f.strip()], "documents_saved": [d.strip() for d in documents_saved.split(",") if d.strip()]}`
         c. Call `client.log_skill_telemetry(skill_name, "in_context_completed", metadata=metadata)`
         d. Return `f"Recorded: {skill_name} executed in-context ({status}). {len(metadata['context_files_written'])} context files, {len(metadata['documents_saved'])} documents."`
         e. Wrap in try/except, return error message on failure (never raise)

    2. Create parity verification script `backend/scripts/verify_parity_156.py`:
       - Purpose: Compare context store writes from in-context vs server-side execution for 5 representative skills
       - The 5 skills to verify (per CUT-04): company-intel, meeting-prep, meeting-processor, daily-brief, sales-collateral
       - This is a REPORTING script, not an automated test. It queries the DB and prints a comparison table.
       - Implementation:
         a. Connect to DB using `get_session_factory()`
         b. For each of the 5 skills, query `context_entries` table:
            - Count entries written by the skill (source LIKE '%{skill_name}%' or metadata contains skill reference)
            - Get average entry length
            - List distinct context file names written to
         c. Query `skill_execution_telemetry` for execution counts by path
         d. Print a formatted table:
            ```
            Skill: company-intel
              Server-side runs: N (from skill_runs table)
              In-context redirects: N (from telemetry, path=redirect_to_in_context)
              In-context completed: N (from telemetry, path=in_context_completed)
              Context files written (server): [list]
              Context files written (in-context): [list from telemetry metadata]
              Parity: PASS/NEEDS_DATA (if no in-context data yet, can't compare)
            ```
         e. Parity criteria (per research pitfall 5): same context file names written to, comparable entry count per run, non-empty content. NOT byte-identical.
         f. Exit with status 0 if all 5 have PASS or NEEDS_DATA, exit 1 only if a skill shows MISMATCH
       - Usage: `cd backend && uv run python scripts/verify_parity_156.py`
       - NOTE: This script will likely show NEEDS_DATA for most skills initially since in-context execution hasn't happened yet. That's expected -- the script exists for ongoing monitoring as adoption grows.
