**Endpoint 1: GET /accounts/{account_id}/timeline (API-04)**

Unified chronological feed combining three source types. Each item has a `type` discriminator.

Query params: `offset` (default 0), `limit` (default 50, max 100)

Implementation approach — 3 separate queries merged in Python (simpler than UNION ALL with differing column shapes):

1. **Outreach activities** (type="outreach"):
   - Select from outreach_activities WHERE account_id matches
   - Map to: `{id, type: "outreach", date: sent_at, title: subject or f"{channel} {direction}", summary: body_preview, status, channel, direction}`

2. **Context entries** (type="context"):
   - Select from context_entries WHERE account_id matches AND deleted_at IS NULL
   - Map to: `{id, type: "context", date: created_at, title: file_name, summary: content[:200], source, confidence}`

3. **Documents** (type="document"):
   - Select from uploaded_files WHERE id IN (select distinct metadata->>'uploaded_file_id' from context_entries where account_id = X and metadata->>'uploaded_file_id' is not null) — OR simpler: join context_entries on account_id to uploaded_files via the context_entry's relationship. Since UploadedFile doesn't have account_id directly, find uploaded_files linked to this account through context_entries that have account_id set. Use: `SELECT DISTINCT uf.* FROM uploaded_files uf JOIN context_entries ce ON ce.tenant_id = uf.tenant_id WHERE ce.account_id = :account_id AND ce.deleted_at IS NULL AND ce.metadata->>'uploaded_file_id' = uf.id::text` — but this is fragile. Simpler: just skip documents in timeline for now if there's no reliable FK. Check if context_entries have metadata with uploaded_file_id. If the linkage is unclear, include only outreach + context entries (the two types with direct account_id FK). Add a TODO comment for document linkage.
   - Actually, UploadedFile has no account_id. For v1, include only outreach activities and context entries in the timeline. These cover the key use cases. Documents can be added later when account_id is added to uploaded_files or a join table exists.

So the timeline merges:
- Outreach activities (type="outreach", date=sent_at)
- Context entries (type="context", date=created_at)

Merge both lists, sort by date DESC, apply offset/limit. Count total from both queries for pagination.

For total count, sum the count of outreach + count of context entries for this account. For the items, fetch both full lists (or use a smarter approach: fetch limit+offset items from each, merge, sort, slice). Since accounts won't have thousands of entries in practice, fetching all and slicing in Python is acceptable for v1.

**Endpoint 2: GET /pulse/ (API-05)**

Pulse signals are computed on-the-fly from CRM data. Each signal has: `{id, type, priority, account_id, account_name, title, detail, created_at}`.

Signal types to compute:

1. **reply_received** (priority: 1 — highest):
   - Find outreach_activities with status="replied" AND created_at within last 7 days
   - Each produces: `{type: "reply_received", title: f"{account.name} replied", detail: subject or channel, priority: 1}`

2. **followup_overdue** (priority: 2):
   - Find accounts where next_action_due < now() AND next_action_due IS NOT NULL AND status IN ("prospect", "engaged")
   - Each produces: `{type: "followup_overdue", title: f"Follow-up overdue: {account.name}", detail: next_action_type or "Follow up needed", priority: 2}`

3. **bump_suggested** (priority: 3):
   - Find accounts with status="prospect" that have outreach_activities but the most recent outreach sent_at is older than 14 days AND no outreach has status="replied"
   - Each produces: `{type: "bump_suggested", title: f"Bump {account.name}?", detail: f"No activity in {days} days", priority: 3}`

Query params: `limit` (default 20, max 50), `type` filter (optional, filter to one signal type)

Implementation:
- Run 3 queries (one per signal type)
- Merge results, sort by priority ASC then created_at DESC
- Apply limit
- Return `{items: [...], total}`

Use `from datetime import datetime, timedelta, timezone` for date math. Use `func.now()` or `datetime.now(timezone.utc)` for comparisons.

For bump_suggested, use a subquery: accounts that are prospects, have at least one outreach activity, but the MAX(sent_at) of their outreach is older than 14 days, and have zero outreach with status="replied". This can be done with:
```python
# Subquery: max sent_at per account for prospects
latest_outreach = (
    select(
        OutreachActivity.account_id,
        func.max(OutreachActivity.sent_at).label("latest_sent"),
    )
    .group_by(OutreachActivity.account_id)
    .subquery()
)
# Subquery: accounts with any reply
replied = (
    select(OutreachActivity.account_id)
    .where(OutreachActivity.status == "replied")
    .distinct()
    .subquery()
)
# Main: prospects with stale outreach and no reply
bump_query = (
    select(Account, latest_outreach.c.latest_sent)
    .join(latest_outreach, Account.id == latest_outreach.c.account_id)
    .outerjoin(replied, Account.id == replied.c.account_id)
    .where(
        Account.status == "prospect",
        replied.c.account_id.is_(None),
        latest_outreach.c.latest_sent < cutoff_14d,
    )
)
```
