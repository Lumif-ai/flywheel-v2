  <action>
Create `backend/src/flywheel/api/outreach.py` with `router = APIRouter(tags=["outreach"])`.

Note: This router uses TWO prefix groups, so set NO prefix on the router itself. Use path prefixes directly on each endpoint instead (e.g., `/accounts/{account_id}/outreach`, `/outreach/{outreach_id}`, `/pipeline/`). Alternatively, use two sub-routers if cleaner — but one router with explicit paths is simpler and matches patterns in the codebase.

**Pydantic models:**
- `OutreachResponse` — id, account_id, contact_id, channel, direction, status, subject, body_preview, sent_at, metadata, created_at
- `CreateOutreachRequest` — channel (required), direction (required), status (default "sent"), subject, body_preview, sent_at (datetime, optional), contact_id (optional UUID), metadata (optional dict)
- `UpdateOutreachRequest` — all optional: status, subject, body_preview, sent_at, contact_id, metadata
- `PipelineItem` — id, name, domain, fit_score, fit_tier, status, last_interaction_at, outreach_count (int), last_outreach_status, days_since_last_outreach (int or None), created_at

**Endpoints (API-03):**

1. `GET /accounts/{account_id}/outreach` — list outreach for account:
   - Verify account exists (404 if not)
   - `offset` (default 0), `limit` (default 50, max 100)
   - Select OutreachActivity where account_id matches, order by sent_at DESC
   - Return paginated response

2. `POST /accounts/{account_id}/outreach` — create outreach activity:
   - Verify account exists (404)
   - If contact_id provided, verify it belongs to this account (400 if not)
   - Create OutreachActivity with tenant_id from user
   - Update account.last_interaction_at to sent_at (or now()) if sent_at > current last_interaction_at
   - Return 201
   - **AUTO-01 trigger:** If status == "replied", call `_maybe_graduate_on_reply(db, account, user)` (defined below)

3. `PATCH /outreach/{outreach_id}` — update outreach activity:
   - Fetch by id, 404 if not found
   - Apply non-None fields
   - **AUTO-01 trigger:** If status field is being updated to "replied":
     - Load the parent account via outreach.account_id
     - If account.status == "prospect", set account.status = "engaged" and account.updated_at = now()
     - Create a ContextEntry logging the graduation: tenant_id from user, user_id from user, file_name="account-events", source="auto:graduation", content=f"Account '{account.name}' auto-graduated from prospect to engaged after reply received on outreach {outreach_id}", date=today, account_id=account.id
   - Commit, return updated outreach dict

4. `GET /pipeline/` — cross-account pipeline view:
   - Select accounts WHERE status = "prospect", order by fit_score DESC NULLS LAST
   - `offset` (default 0), `limit` (default 50, max 100)
   - For each account, compute: outreach_count (count of outreach_activities), last_outreach_status (status of most recent outreach by sent_at), days_since_last_outreach (days between now and most recent outreach sent_at, or None if no outreach)
   - Use a subquery or left join for outreach stats to avoid N+1
   - Return paginated PipelineItem list

5. `POST /accounts/{account_id}/graduate` — manual graduation:
   - Fetch account, 404 if not found
   - If account.status != "prospect", return 400 "Account is not a prospect"
   - Set status = "engaged", updated_at = now()
   - Create ContextEntry: file_name="account-events", source="manual:graduation", content=f"Account '{account.name}' manually graduated from prospect to engaged", date=today, account_id=account.id, tenant_id and user_id from token
   - Commit, return updated account dict

**Helper function:**
- `_outreach_to_dict(o: OutreachActivity) -> dict` — serialization helper

**Important implementation detail for AUTO-01:** The graduation logic (check prospect status, update to engaged, log context entry) is shared between PATCH /outreach/{id} (auto) and POST /accounts/{id}/graduate (manual). Extract into a shared helper `_graduate_account(db, account, user, source_label)` that both call.
