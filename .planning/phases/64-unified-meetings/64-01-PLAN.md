**calendar_sync.py changes:**

1. Replace `upsert_meeting_work_item()` with `upsert_meeting_row()`:
   - Change imports: remove `WorkItem, MeetingClassification, SuggestionDismissal`, add `Meeting`
   - Remove `from flywheel.services.meeting_classifier import classify_meeting` (classification happens during intelligence processing, not at calendar sync time)
   - Lookup by `calendar_event_id` (not external_id): `Meeting.calendar_event_id == event["id"]`
   - **Skip guard:** If existing row has `granola_note_id` set, return immediately (Granola data is richer)
   - Handle cancelled events: if `event.status == "cancelled"` and existing Meeting found, set `processing_status = "cancelled"`. If existing has `granola_note_id` set, do NOT cancel.
   - Parse attendees into Meeting-compatible format: `[{"email": ..., "name": ..., "is_external": ...}]`. Mark `is_external` based on `not a.get("self", False) and not a.get("organizer", False)`.
   - Set `provider = "google-calendar"`, `external_id = f"gcal:{event['id']}"`, `calendar_event_id = event["id"]`
   - Set `processing_status = "scheduled"` (not WorkItem's "upcoming")
   - Map `event["location"]` -> `location`, `event["description"]` -> `description`
   - Remove all MeetingClassification auto-record logic (the entire try/except block after db.flush)

2. Update `sync_calendar()`: change the call from `upsert_meeting_work_item` to `upsert_meeting_row`.

3. **RLS context in calendar_sync_loop():** Before calling `sync_calendar()` for each integration, set RLS context on the session:
   ```python
   await session.execute(text("SET LOCAL app.tenant_id = :tid"), {"tid": str(integration.tenant_id)})
   await session.execute(text("SET LOCAL app.user_id = :uid"), {"uid": str(integration.user_id)})
   ```
   Import `text` from `sqlalchemy`. This is critical because Meeting has split-visibility RLS (owner_write policy requires user_id match).

4. Keep `get_meeting_prep_suggestions()` unchanged for now (Plan 02 migrates it).

**meetings.py changes:**

5. Add `_find_matching_scheduled()` helper function for Granola fuzzy dedup:
   - Query Meeting rows with `processing_status == "scheduled"` and `meeting_date` within +/-30 minutes of the incoming Granola meeting
   - Title match: case-insensitive contains (either direction)
   - Attendee overlap: at least one email in common
   - Return first matching candidate or None

6. Update `sync_meetings()` endpoint: After the existing dedup-by-external_id check, for each new Granola meeting, call `_find_matching_scheduled()`. If a match is found:
   - Enrich the existing row: set `granola_note_id = raw.external_id`, `ai_summary = raw.ai_summary`, `duration_mins = raw.duration_mins`, `processing_status = "recorded"`
   - Do NOT insert a new row
   - Count as "synced" (since the match is a new enrichment)
   If no match, insert as before with `provider="granola"`, `granola_note_id=raw.external_id`.

7. Update `process_pending_meetings()` WHERE clause: change `Meeting.processing_status == "pending"` to `Meeting.processing_status.in_(["pending", "recorded"])`. This implements UNI-03's requirement that recorded meetings (Granola-matched calendar events) get processed too.

8. Update `process_meeting()` guard: add "recorded" and "scheduled" to the allowed states for reprocessing. Change the guard to: `if meeting.processing_status in ("processing", "complete"): raise 409`.
