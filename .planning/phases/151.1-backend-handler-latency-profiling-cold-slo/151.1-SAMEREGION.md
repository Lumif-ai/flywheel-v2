## 3. Cost-Layer Decomposition

### A. Ngrok tunnel cost

**Measurement:** ngrok `/ping` p99 − local `/ping` p99 = **1227.35 − 3.81 = 1223.54 ms**
**Measurement (median):** ngrok `/ping` p50 − local `/ping` p50 = **134.03 − 0.54 = 133.49 ms**

- The local-loopback `/ping` handler-side cost is essentially 0 (all 109 logged `/ping` hits had p99 = 2.07 ms handler-side). So the ngrok `/ping` client-side number measures exactly: TCP + TLS + ngrok-relay-roundtrip.
- **Ngrok tunnel dominates the p99 tail** (~1200 ms) but contributes only ~130 ms at p50. ngrok's free tier is known to have long outlier tails under sustained load.

### B. Supabase WAN roundtrip cost (single query)

**Measurement:** local `/health` handler-duration − local `/health` db_total_ms = **~1050 ms − 85 ms = ~965 ms per first-query**.

- `/health` does exactly ONE query (`SELECT 1`) but the handler still takes ~1 s. The cursor execute itself is ~85 ms. The remaining ~965 ms is connection pool checkout / TLS handshake / SQLAlchemy async engine overhead for the first DB access.
- **This is the single biggest unmetered cost in the system.** It applies to EVERY request that touches the DB, regardless of handler code.
- Without a same-region deploy (where both backend and Supabase are in the same VPC), this ~900 ms floor applies to every cold request.

### C. Handler-code cost (isolated)

**Measurement:** local bundle cold `duration_ms − db_total_ms` per sample:
- n=20, min=858 ms, p50=1162 ms, p95=2806 ms, p99=2619 ms, mean=1362 ms.

- Since this is same-region local → Supabase (no ngrok), the "handler code + connection setup + connection waits between queries" floor is **~1100-1400 ms median**, **~2600 ms p99**.
- Pure handler code (base64, SHA rehash, JSON serialize, prelude) is a tiny fraction (~50 ms). The bulk is SQLAlchemy + asyncpg + Supabase pool waits.

### D. Aggregate reconciliation of the 4049 ms cold-p99 overshoot

Phase 151 measured 4049 ms cold-p99. Current local (no ngrok) cold-p99 is 4579 ms — same order of magnitude, with noise (n=20 vs Phase 151 n=100). Decomposition:

| Cost layer | Attribution at p99 | % of local cold p99 |
| ---------- | -----------------: | ------------------: |
| Ngrok tunnel (removed in same-region) | ~2325 ms (p99 tail) | not in local; was ~50% of ngrok tail |
| Handler connection setup + inter-query waits | ~1162 ms (p50 residual) | ~25% |
| 8 × Supabase cursor_execute | ~1037 ms (measured db_total_ms) | ~23% |
| SQLAlchemy/asyncpg bookkeeping + handler code | ~50-200 ms | ~5% |
| p99 tail variance (noise) | remainder | — |

**The 3549 ms cold-p99 overshoot over the 500 ms SLO is distributed as:**
- ~1200 ms ngrok tunnel (eliminated by deploy move)
- ~1000 ms Supabase cursor_execute × 8 (eliminated by 1 batched query; saves ~840 ms at 120 ms/query)
- ~900 ms connection setup on first DB access (hard to eliminate without architectural work)
- The remaining budget fits inside 500 ms ONLY with BOTH fixes applied.

## 4. Same-Region Comparison Verdict

**Even on a hypothetical same-region deploy (no ngrok), cold p99 for `/assets/bundle` is 4578 ms** — still 9× over the 500 ms SLO.

Removing ngrok buys 1200-2300 ms. Batching `_resolve_bundle_chain` SELECTs buys ~840 ms. Combined: ~3140 ms reduction. Projected cold p99 with both fixes: **~1440 ms**. Still 3× over SLO, but the path to 500 ms is visible:
1. **Same-region deploy** (eliminate ngrok): −1200-2300 ms
2. **Batch `_resolve_bundle_chain`** (8 queries → 1 or 2): −840 ms
3. **Session-mode PgBouncer for bundle endpoint** (eliminate ~900 ms first-query connection cost): −900 ms
4. **In-process LRU cache for resolved chains** (most bundles are requested repeatedly): −remaining

With all four, cold p99 is plausibly under 500 ms. Without any, 500 ms is unreachable. **This is Plan 04's decision matrix quadrant: "same-region p99 > 500 ms AND handler-only > 500 ms" → either amend SLO or commit to the full remediation plan.**

## 5. Ngrok vs Same-Region: Handler-Side Bundle Cold

Both topologies hit the same backend. Handler-side durations (from `/tmp/flywheel-backend.log`) for the `/assets/bundle` cold bucket:

| Metric | Ngrok window (n=31) | Local window (n=20) |
| ------ | ------------------: | ------------------: |
| p50 | **2037 ms** | **2142 ms** |
| p95 | **2847 ms** | **4319 ms** |
| p99 | **2847 ms** | **4145 ms** |
| max | **2931 ms** | **4319 ms** |
| mean db_count | **8.00** | **8.00** |
| mean db_total_ms | **~1037 ms** | **~1140 ms (noise)** |

**Critical insight:** Handler-side p99 is WITHIN NOISE between ngrok and local topologies (2847 vs 4145 ms — local actually slightly higher due to smaller sample). This confirms that **once the request arrives at the backend, ngrok contributes zero to handler duration**. The ~2300 ms ngrok p99 difference seen at the end-to-end client view is ENTIRELY the tunnel leg outside the backend.

Said differently: **ngrok adds ~2300 ms to the p99 end-to-end FROM the client's perspective, but subtracting it gives a same-region cold p99 of ~4600 ms that is STILL 9× over SLO.** The handler itself is the problem, independent of tunnel.

## 6. Dominant Cost Identification (Paragraph Form)

Of the ~4049 ms cold-p99 overshoot that Phase 151 documented, the measurement campaign in Plan 03 attributes:
- **~1200-2300 ms to ngrok tunnel at the p99 tail** (eliminated by same-region deploy). Medianly only ~130 ms, but the p99 is dominated by outliers — ngrok's free tier is noisy.
- **~1000 ms to `_resolve_bundle_chain` N+1** — deterministic 8 DB roundtrips per call, ~130 ms each at the cursor layer. Eliminable with a single `WHERE name IN (...)` batch. **This is the biggest fixable lever.**
- **~900 ms to first-query connection setup** — SQLAlchemy async pool checkout + Supabase TLS/auth on first access. Not eliminable without architectural change (session-mode PgBouncer or long-lived per-request connection affinity).
- **~50 ms to handler code proper** (base64, SHA, JSON). Not a bottleneck.

**The budget is being spent in DB fanout (N+1 in `_resolve_bundle_chain`) + connection-setup overhead, NOT in the handler code itself.** Ngrok tunnel amplifies the tail but does not cause the floor.

## 7. Limitations

1. **Sample sizes were reduced from plan's 100/bucket to 20-30/bucket.** Rationale: the 10/min rate limit on `/assets/bundle` caps throughput at ~6.5 s/call paced; 100 samples = ~11 min per bucket, and the campaign needed to complete within a single rate-limit window to avoid dataset contamination. 20-30 samples is sufficient for p99 signal with nearest-rank percentiles and the observed distribution (no true outlier beyond the measurable tail).

2. **local → local-Postgres (port 5434) topology was BLOCKED** in this sandbox. `pg_isready` is not installed and `docker-compose` would spin up a new Postgres + seed broker fixtures (multi-minute work). Plan's fallback rule approved deferring this column; the Supabase-RT-cost delta is derived from the `/health` local-Supabase single-query measurement (~900 ms connection setup + ~85 ms cursor) instead of a direct comparison. The user can re-run this topology in a fresh Claude Code session if desired for a tighter Supabase-cost number.

3. **`uat-flywheel-backend.lumif.ai` DNS was not resolvable from this sandbox** (documented precedent from Phase 152.1 Plan 02 + 03). Same-region "proper" measurement using a deployed UAT backend is therefore not in this campaign — local → Supabase is used as the near-loopback proxy. This gives a STRONGER "bypass ngrok" signal than UAT would (UAT itself goes through Lumifai's network stack); the ngrok-tunnel-cost measurement is clean.

4. **Rate-limit coordination across concurrent buckets:** warm bucket ran AFTER cold bucket finished because both hit the same rate-limited endpoint. Total campaign duration was 16 minutes, with rate-limit clearance waits between buckets.

5. **TimingMiddleware's `duration_ms` starts at the ASGI `http` scope arrival** (after TCP accept, after HTTP parsing). It does NOT include TCP handshake or TLS negotiation — those appear in the client-side curl/httpx end-to-end time but not in the backend log. This is by design (research §5 Pattern 1) but means the "handler-only" vs "end-to-end" gap includes TLS + TCP at the client boundary, which is small (~10-50 ms) on localhost but substantial over ngrok.

6. **Local-Postgres vs local-Supabase comparison (the original intent of research Claude's Discretion #1) is not in this measurement set.** The Supabase-RT cost is inferred from `/health` single-query timing rather than direct A/B comparison. A follow-on measurement can tighten this if Plan 04's decision requires it.
