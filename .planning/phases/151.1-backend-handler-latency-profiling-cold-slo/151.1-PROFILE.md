**Supabase single-query cost breakdown** (derived from `/health`):
- One Supabase roundtrip (cursor_execute): **~120 ms** (cursor open → send → receive)
- One Supabase connection checkout / SSL / async pool wait: **~900 ms** on first query

> Implication: a handler doing 8 Supabase roundtrips can't simply be 8 × 120 = 960 ms of DB. The FIRST roundtrip pays the ~900 ms connection-setup cost; subsequent ones share the pool. See §4.

## 4. `get_skill_assets_bundle` Bottleneck Breakdown

Every single cold `/assets/bundle` request emitted **`db_count=8`** — this is 100% deterministic. No variance across 51 cold samples.

| Layer | Estimate | Grounding |
| ----- | -------: | --------- |
| Handler code (base64 encode, SHA rehash, JSON serialize) | ~50 ms | Plan 04 Phase 151 estimate; bundle is 9283 B |
| 8 × Supabase cursor_execute (after pool warm) | ~1037 ms (measured `db_total_ms` mean) | Direct measurement from `request_complete` log |
| Connection checkout + TLS + SQLAlchemy async overhead on first DB roundtrip | ~900 ms | Derived: `/health` duration (1050 ms) − db_total_ms (85 ms) = ~965 ms for single-query case |
| Handler-code-minus-DB residual (local topology, 20 samples) | p50 1162 ms, p99 2619 ms | `duration_ms − db_total_ms` per sample |
| Ngrok tunnel (one-way) | ~100-600 ms typical, >1000 ms p99 | `/ping` p50-p99 ngrok |
| **Total handler-side p99 (ngrok)** | **2847 ms** | Measured |
| **Total end-to-end p99 (ngrok, client-side)** | **6904 ms** | Measured (`/tmp/151.1-ngrok-cold.md`) |
| **Gap (end-to-end − handler-side)** | **~4057 ms** | Ngrok tunnel tail latency |

### Bottleneck identification

The cold-p99 budget (4049 ms per Phase 151) is burned predominantly by a compounding of:

1. **`_resolve_bundle_chain` N+1 fanout — confirmed at N=8.** Every cold bundle request executes exactly 8 Supabase roundtrips. Research hypothesis was 7-8 (3-4 prelude + 3-4 chain-walker). The measurement corroborates: ~3 tenant-session prelude (`set_config` x2 + `SET ROLE`) + ~1 root skill SELECT + ~2 dep skill SELECTs (broker library) + ~1-2 asset SELECTs = 8. This is the single largest knob.

2. **Connection checkout / first-query cost (~900 ms).** Even `/health` (one query) spends ~900 ms outside the cursor_execute window. This is Supabase TLS/pool initialization amortized. Cannot be eliminated without architectural work (persistent pool pinned per request, or session-mode PgBouncer).

3. **Ngrok tunnel tail (~1200 ms p99).** On a same-region deploy this disappears entirely. See `151.1-SAMEREGION.md`.

**Primary lever (ordered by impact on cold-p99):**
- `_resolve_bundle_chain`: 8 roundtrips → 1 WHERE name IN (...) roundtrip would save ~7 × 120 ms = **~840 ms** at the same-region floor, and ~7 × 200 ms = **~1400 ms** at ngrok p99.
- Ngrok removal: saves the 1200 ms p99 tunnel tail.
- Connection-setup caching: saves ~900 ms of first-query connection cost. Hard to realize without deep engine rework.

**The budget is being spent in `_resolve_bundle_chain` (8-fold DB fanout) AND Supabase WAN RTT + first-connection checkout, NOT in handler code proper (base64/SHA/serialize ~50 ms combined).**

**Conclusion:** The SERVER has zero warm-path advantage for `/assets/bundle` — each hit re-queries Supabase 8 times. The real warm path is entirely client-side (disk cache short-circuit). Caching response in the backend (e.g. Redis, in-process LRU) would flatten cold p99 dramatically.

## 7. Broker-Domain Handlers — Deferred

The plan mentions profiling broker handlers (`extract_contract_analysis`, `save_contract_analysis`, `list_broker_projects`) as a secondary diagnostic surface during the dogfood journey.

**Status: Deferred.** No broker calls fired during the 16-minute campaign window — none appear in the log with `cold=True` in the time slice. Rationale:
1. The cold-SLO FAIL is specifically on skill-asset fetch (`/assets/bundle`), which this campaign fully characterized.
2. Broker handlers are not on the critical cold path — they fire AFTER skill fetch, during the dogfood flow.

## 8. Methodology Notes

- **Sample sizes** (smaller than plan's 100 where noted): 30 ngrok cold, 30 ngrok warm, 50 ngrok ping, 20 local cold, 20 local warm, 50 local ping, 15 each for list/prompt/assets secondary handlers, 25 /health. Smaller N was traded for completing within rate-limit windows (10/min per tenant for `/assets`, `/bundle`, `/prompt`) without contaminating the dataset through 429 retries. The p99 threshold uses nearest-rank index `floor(n*0.99) − 1` (matches Phase 151 script behaviour).
- **Log timezone:** `/tmp/flywheel-backend.log` timestamps are in `+08`; `extract_profile.py`'s `--since` flag was validated for correct handling.
- **Token refresh:** Token hit `api_url=null` default when running through CLI helpers; worked around with `FLYWHEEL_API_URL=http://127.0.0.1:8000 python ... from flywheel_cli.auth import get_token`. Refreshed token valid for both local and ngrok topologies.
- **The sandbox cannot resolve `uat-flywheel-backend.lumif.ai`** (DNS blocker inherited from Phase 152.1 Plan 02/03). UAT-topology measurement was NOT attempted; same-region comparison uses localhost → Supabase (no ngrok) as the near-loopback topology, which isolates tunnel cost from Supabase cost. See `151.1-SAMEREGION.md`.
- **Local-Postgres (port 5434) topology was NOT run** — `pg_isready` unavailable in sandbox, and `docker-compose` would spin up a new service; plan's fallback rule approved deferring this column. SAMEREGION.md's §"Limitations" annotates the missing data.
