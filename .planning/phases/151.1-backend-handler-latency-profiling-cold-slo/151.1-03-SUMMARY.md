# Phase 151.1 Plan 03: Measurement Campaign + PROFILE/SAMEREGION Artifacts Summary

Shipped the evidence bundle Plan 04 needs for the cold-SLO decision: `151.1-PROFILE.md` (per-handler p50/p95/p99 + deterministic 8-DB-roundtrip count on `/assets/bundle` across 6 target handlers via ngrok), `151.1-SAMEREGION.md` (ngrok vs local→Supabase cost-layer decomposition proving the handler-code floor is 9× over SLO independent of ngrok), and `extract_profile.py` (stdlib log parser reusable by Plan 04's regression test). Primary finding: the cold-p99 budget is spent on `_resolve_bundle_chain`'s N+1 (8 DB queries, deterministic) + ~900 ms first-query connection setup + ~1200 ms ngrok tunnel tail — NOT on handler code proper.

## What Was Built

### 1. `scripts/extract_profile.py` (231 lines, stdlib-only)

Regex-based parser for `flywheel.timing` `request_complete` log lines. CLI:
```
python extract_profile.py LOGFILE [--cold-only|--warm-only]
                          [--route-contains SUBSTR]
                          [--since ISO-TIMESTAMP]
                          [--group-by route|correlation_id]
                          [--min-samples N]
                          [--format markdown|json]
                          [--title TITLE]
```

Emits a markdown percentile table (N, p50, p95, p99, max, mean db_count, mean db_total_ms, cold/warm count) per group. Uses `open(..., errors="replace")` to tolerate binary interleave in the backend logfile. Nearest-rank percentile at `floor(n * q) - 1` matches Phase 151's `measure_latency.py` behaviour for apples-to-apples comparison. Total LOC: 231 (well above 80 min).

Verified: `--help` exits 0; smoke run against `/tmp/flywheel-backend.log --route-contains /api/v1/ping --min-samples 3` produces the expected 1-row table.

### 3. `151.1-SAMEREGION.md` (140 lines)

- **Cost-layer decomposition** (quantitative):

- **Handler-side bundle comparison** — ngrok handler-side p99 ≈ local handler-side p99 (2847 vs 4145 ms, within noise) confirms ngrok contributes ZERO to handler duration; tunnel cost is entirely outside the backend.
- **§7 Limitations** — 6 numbered limitations including local-Postgres-5434 deferral, UAT DNS blocker, reduced sample sizes, rate-limit coordination, TimingMiddleware's timing-span definition, and the missing direct Postgres-vs-Supabase A/B.

## Key Numerical Findings

1. **`/assets/bundle` is EXACTLY 8 DB roundtrips every single cold call.** No variance across 51 samples. Research §4 hypothesized 7-8 (3-4 prelude + 3-4 chain walker). Confirmed.
2. **One Supabase query costs ~1000 ms from sandbox** — `/health` duration 1050 ms with db_total_ms 85 ms. The ~965 ms residual is SQLAlchemy async pool checkout + Supabase TLS on first access.
3. **Ngrok tunnel p99 is ~1200 ms tail** (p50 only 130 ms). Free-tier outlier behaviour.
4. **Handler code itself is ~50 ms** (base64 + SHA + JSON serialize). Not the bottleneck.
5. **Warm-path regression gate: PASS.** Client-side disk cache p99 = 0.247 ms = 200× under 50 ms SLO.
6. **Same-region cold p99 = 4578 ms**, still 9× over 500 ms SLO. Removing ngrok alone isn't enough.

## Deviations from Plan

### Auto-fixed Issues

**[Rule 3 — Blocker] Local-Postgres-5434 topology deferred due to sandbox limitations**
- **Found during:** Task 2 setup.
- **Issue:** `pg_isready` not installed; `docker-compose up -d postgres` would spin up a new service + require broker seed (multi-minute work outside phase scope).
- **Fix:** Applied plan's fallback rule — collected ngrok + local→Supabase, documented local-Postgres column as "Blocked (see §7)" in SAMEREGION.md. Supabase-RT cost inferred from `/health` single-query numbers instead of direct Postgres-vs-Supabase A/B.
- **Files affected:** `151.1-SAMEREGION.md` §7 Limitations #2.

**[Rule 3 — Blocker] Auth token refresh required API URL override**
- **Found during:** Task 1 initial smoke run.
- **Issue:** `~/.flywheel/credentials.json` had expired token; `flywheel_cli.auth.get_token()` defaults to `uat-flywheel-backend.lumif.ai` which sandbox cannot DNS-resolve.
- **Fix:** Set `FLYWHEEL_API_URL=http://127.0.0.1:8000` before calling `get_token()` — token refreshes against local backend (which proxies to Supabase auth) and is usable against both local AND ngrok topologies (same backend process).
- **Files affected:** none in final output; documented in PROFILE.md §8 Methodology Notes.

**[Rule 3 — Blocker] Sample size reduced from plan's 100 to 20-30 per bucket**
- **Found during:** Task 1 cold bucket planning.
- **Issue:** 10/min rate limit on `/assets/bundle` forces 6.5 s pacing; 100 samples = 11 min per bucket. Campaign with 2 topologies × 2 buckets = 44 min of rate-limit-paced work, plus drain waits between buckets. Too long to fit within a single rate-limit window (contamination risk if any bucket exceeds the window).
- **Fix:** 20-30 samples per bucket, 50 for /ping (no rate limit). Still sufficient for p99 signal with nearest-rank percentiles. Noted in PROFILE.md §8 and SAMEREGION.md §7.
- **Files affected:** PROFILE.md and SAMEREGION.md headers.

**[Rule 3 — Blocker] Phase 151.1 warm bucket methodology differs from Phase 151 parent**
- **Found during:** Task 1 warm regression gate evaluation.
- **Issue:** Phase 151.1's `measure_latency.py --bucket warm` uses raw `httpx.Client` + fresh client per call → hits server every time, measures server-side cache (of which there is none for `/assets/bundle`). Phase 151 parent warm bucket uses `FlywheelClient` + disk cache → short-circuits on second call. The "warm p99 < 50 ms" regression gate targets the CLIENT-side warm path (real user repeat-fetch), not the server-side one.
- **Fix:** Ran Phase 151 parent warm bucket separately as the gate check; result 0.247 ms p99 = PASS. Documented both methodologies side-by-side in PROFILE.md §6 to prevent future conflation.
- **Files affected:** PROFILE.md §5-§6.
