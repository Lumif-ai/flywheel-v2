2. **[Rule 3 — Blocking] ContextVar reset-per-request invalidated post-request read.**
   - **Found during:** Task 2 design review (after first green test, ran diagnostic to confirm db_count was non-zero — it was 0 post-request, because TimingMiddleware resets the ContextVar at request start).
   - **Issue:** Original test used `db_count_cv.set(0)` pre-request then `db_count_cv.get()` post-request. ContextVars are scoped to the request coroutine; after TestClient returns, the test-scope read sees the pre-request value (0), not the post-request increment. Test passed trivially at `0 <= 10`.
   - **Fix:** Refactored to parse `db_count=N` from TimingMiddleware's `request_complete` log line via pytest `caplog.at_level(INFO, "flywheel.timing")`. Added a sanity assertion `count > 0` to prevent silent re-regression if TimingMiddleware ever gets removed. Added module docstring section "How we read the count" explaining the observation surface parity with production's `extract_profile.py`.
   - **Files modified:** `backend/src/tests/test_bundle_handler_db_count.py` (imports + helper `_extract_db_count_from_log` + both tests).
   - **Commit:** Part of the per-plan commit.
