## 2. Decision Matrix Applied

Per research §6 decision matrix, the inputs are:

- Same-region cold p99 = **4578 ms** → well above the 500 ms threshold.
- Handler-only cold p99 = **4145 ms** (same-region) / **2847 ms** (ngrok) → well above 500 ms in BOTH topologies.

**Quadrant hit:** *"same-region cold p99 > 500 ms AND handler-only cold p99 > 500 ms"* → matrix says **"handler code + N+1 floor is above budget; SLO must be amended OR `_resolve_bundle_chain` must be batch-rewritten (alongside at least 2 other remediations) to make 500 ms achievable."**

This is not a borderline call. Same-region cold p99 is **9.2× over** the SLO *without* ngrok in the picture. Ngrok's tunnel tail (~1200 ms p99) is responsible for roughly half the ngrok-vs-same-region gap (2326 ms delta; see SAMEREGION.md §5 "critical insight"), but removing it still leaves 4578 ms of handler + DB cost — well above any achievable-without-work budget.

## 3. The Two Viable Paths

### Option A (APPROVED 2026-04-20): AMEND the SLO now + ship remediation as follow-up phases

Use the research §6 amendment template. Replace ROADMAP SC5 wording with:

> **SC5 (amended 2026-04-20 via Phase 151.1):** Measured against live ngrok + Supabase, p99 cold first-fetch latency < **3500 ms** *on ngrok tunneled dev topology*, with a corresponding SLO of < **1500 ms** on same-region deploy (documented gap: ~2300 ms p99 attributable to ngrok tunnel tail latency, verified by Phase 151.1 same-region comparison measurement in `151.1-SAMEREGION.md`). p99 warm-cache latency < 50 ms (unchanged, 0.247 ms measured). Numbers recorded in `.planning/phases/151.1-backend-handler-latency-profiling-cold-slo/151.1-SLO-DECISION.md`.

**Threshold rationale:**
- Ngrok 3500 ms allows ~20% headroom above the measured 2847 ms handler-only p99 + the observed ngrok tail. Today's measurement at the p99 percentile is 6904 ms; 3500 ms is tighter than the measurement (i.e. we are NOT tuning to just-pass), forcing the `_resolve_bundle_chain` batch fix to land before this SLO can pass regression.
- Same-region 1500 ms allows ~5% headroom above the projected post-batch-fix floor (4578 ms − 840 ms batch save − 900 ms connection-setup-if-session-mode-PgBouncer = ~2840 ms worst case; ~1440 ms best case per SAMEREGION.md §4 projection). 1500 ms forces BOTH batch-fix AND session-mode PgBouncer to land, but does not require backend LRU caching. Honest commitment.

**Why AMEND is recommended:**
1. **The 500 ms target was stale from the start.** It was set against a hypothetical 160 KB bundle (`151-LATENCY.md` opening note); the real bundle is 9.3 KB (17× smaller). The threshold was a guess, not a measurement.
2. **Reaching 500 ms requires all 4 fixes landing together:** same-region deploy + batch query + session-mode PgBouncer + backend LRU. Any one missing = SLO unreachable. Research SAMEREGION.md §4 explicitly states this. Committing to all four up front commits 3–4 phases of backend work on speculative numbers (the post-fix p99 is a projection, not a measurement).
3. **Amendment with documented topology gap is honest;** keeping a demonstrably unachievable SLO ships a bad user commitment. Research §6 warns against the converse ("do not tune the SLO to match the measurement") BUT that rule applies to tuning the NEW SLO to just-pass — here we're tuning to an honest defensible number with headroom that still forces the most valuable fix (batch query).

**Follow-up remediation roadmap (if Option A chosen):**

| Follow-on phase | Work item | Projected impact | Status |
| --- | --- | --- | --- |
| Phase 152+ | Batch `_resolve_bundle_chain` SELECTs into single `WHERE name IN (...)` — collapses 8 roundtrips → 1–2 | −840 ms at same-region floor; −1400 ms at ngrok p99 | Named, not yet scheduled |
| Phase 152+ | Move backend off ngrok to same-region Supabase deploy (UAT / prod) | −1200 ms p99 tunnel tail | Named, depends on infra/deploy work |
| Phase 152+ | Switch to session-mode PgBouncer (or long-lived per-request connection affinity) for `/assets/bundle` | −900 ms first-query connection-setup cost | Named, requires pooler mode change |
| Phase 152+ (optional) | In-process LRU cache for resolved bundle chains keyed by `(skill_name, tenant_id)` | Remaining budget to reach 500 ms | Named, optional — only needed if session-mode + batch + same-region don't close the gap |

After batch-fix ships, re-run `measure_latency.py --bucket cold --n 25` against same-region deploy and append the post-fix p99 to this document as an evidence-bearing amendment to SC5 (further tightening).

### Option B (NOT SELECTED — surfaced for completeness): KEEP 500 ms SLO + commit to full 4-item remediation stack NOW

If the user prefers to lock the 500 ms user-experience commitment to the roadmap before the remediation work ships (accepting that Phase 151 SC5 stays `human_needed` until the remediation completes):

1. Keep SC5 at `< 500 ms` verbatim.
2. Annotate ROADMAP SC5 with: `[Validated 2026-04-20 via Phase 151.1 — cold p99 currently 4049 ms (ngrok) / 4578 ms (same-region); SLO achievability requires remediation phases 152+ per 151.1-SLO-DECISION.md.]`
3. Schedule all 4 remediation items as named phases (not "someday"):
   - **Phase 152.2 (proposed):** Batch `_resolve_bundle_chain` — single `WHERE name IN (...)` SELECT for chain + assets. ~30 min of actual code; separate phase for measurement + regression discipline.
   - **Phase 153 (proposed):** Same-region UAT deploy. Infra work.
   - **Phase 154 (proposed):** Session-mode PgBouncer for `/assets/bundle`. Pooling mode split.
   - **Phase 155 (proposed):** Backend LRU cache for resolved chains. Only if 152.2+153+154 combined doesn't hit 500 ms p99 on re-measurement.
4. Phase 151 stays `human_needed` until all 4 remediation phases ship AND a fresh `measure_latency.py` run shows p99 < 500 ms.

**Why Option B is the alternative, not the recommendation:**
- Ships a commitment (500 ms) whose achievability is projected, not measured — if any of the 4 fixes underperforms its projection, the SLO is broken and must be re-amended anyway.
- Keeps Phase 151 SC5 `human_needed` for 3–4 additional phases of work (estimated weeks of calendar time), delaying the UAT gate closure.
- Provides no value over Option A's AMEND + remediation roadmap except "the roadmap text says 500 ms right now," which is a political artifact, not an engineering one.

## 4. Research §9 Claude's Discretion Resolutions

1. **#1 Same-region deploy choice** (which topology to compare against ngrok). **Resolution:** Ran local→prod-Supabase in Plan 03 (the near-loopback proxy — no ngrok in the loop, Supabase WAN still traversed). Local→local-Postgres-5434 was DEFERRED due to sandbox blockers (`pg_isready` missing, `docker-compose` spin-up out of scope); Supabase-RT cost was inferred from the `/health` single-query measurement instead of direct A/B. Evidence in `151.1-SAMEREGION.md` §7 Limitations. This gives a strong ngrok-cost isolation; Supabase-specific cost is inferred rather than A/B'd — fine for a diagnose-and-decide phase.

2. **#2 `_resolve_bundle_chain` batch-fix scope** (include in this phase or defer). **Resolution:** db_count = 8 deterministic across 51 samples confirms N+1 at N≥4 (research bar for including the fix in-phase). The trivial `WHERE name IN (...)` rewrite would save ~840 ms at the same-region floor. **Planner recommendation: DEFER** to a named follow-up phase (Phase 152+) for phase-boundary discipline — this is a diagnose-and-decide phase, not a fix phase. Including the fix would conflate diagnosis with remediation and blur Phase 151's UAT gate scope. The regression test in Task 2 guards against silent worsening; the actual improvement lives in a named follow-up. User can override at Plan 04 Task 4 checkpoint via "add batch fix" resume-signal, in which case a Plan 05 is created inline.

3. **#3 Tenant-session prelude cost** (3–4 `set_config` + `SET ROLE` roundtrips per request; fix or defer). **Resolution:** Measured and reported only in this phase. Per SAMEREGION.md §3.B, the first-query connection setup (~900 ms) is the dominant tenant-session cost, not the `set_config` itself (individual SQL execs are ~85 ms each; 3 × 85 ≈ 255 ms = ~6% of the 4578 ms same-region cold p99). Combining statements into a single `execute` call (research Option B) is beneath the threshold (<30% of cold latency) per research's deferral rule. The real win is session-mode PgBouncer (Option C), which eliminates the 900 ms connection-setup cost entirely and is listed in both Option A's remediation roadmap and Option B's Phase 154 proposal.

4. **#4 `/ping` endpoint clean network baseline** (add or skip). **Resolution:** Added in Plan 02 (`backend/src/flywheel/api/health.py` +9 LOC, 2 tests in `test_ping_endpoint.py`). Produced the clean network baseline used in SAMEREGION.md §3.A "Ngrok tunnel cost" decomposition. Without `/ping`, the tunnel-vs-handler cost split would be unfalsifiable (research Pitfall 8: `/health` is CONTAMINATED with `SELECT 1`). Task closed.
