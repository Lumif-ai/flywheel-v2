Create `backend/src/flywheel/engines/email_voice_updater.py` with:

1. `_compute_diff_summary(original: str, edited: str) -> str` — uses `difflib.unified_diff` to produce a capped (50 lines) human-readable diff summary. Returns "No changes detected." if identical.

2. `update_from_edit(db: AsyncSession, tenant_id: UUID, user_id: UUID, original_body: str, edited_body: str) -> bool` — the main entry point:
   - Load current EmailVoiceProfile for (tenant_id, user_id). If None, log warning and return False (no profile to update).
   - Compute diff summary via `_compute_diff_summary`.
   - If diff is "No changes detected.", return False.
   - Build current profile JSON from the ORM row: `{"tone": profile.tone, "avg_length": profile.avg_length, "sign_off": profile.sign_off, "phrases": profile.phrases}`.
   - Call Haiku (`claude-haiku-4-5-20251001`) with a voice update prompt (see research Example 3 for prompt structure). System prompt tells Haiku to analyze the diff and return ONLY changed fields as JSON. Allowed return fields: `tone`, `avg_length`, `sign_off`, `phrases_to_add`, `phrases_to_remove`.
   - Parse the JSON response (with regex fallback, same pattern as `_extract_voice_profile` in gmail_sync.py).
   - Merge updates into the existing profile:
     - `tone`: replace if returned
     - `avg_length`: running average: `(current * samples + new) / (samples + 1)`
     - `sign_off`: replace if returned
     - `phrases`: add `phrases_to_add`, remove `phrases_to_remove`, cap at 10 items, deduplicate (case-insensitive)
     - `samples_analyzed`: increment by 1
   - Update the profile row via UPDATE (not upsert — row must exist). Set `updated_at = now(UTC)`.
   - Commit and return True.
   - Wrap entire function in try/except — log error with tenant_id and user_id only (no PII), return False on failure. Use `settings.flywheel_subsidy_api_key` for the Haiku call (same as scorer/drafter pattern).

Create `backend/src/flywheel/engines/email_dismiss_tracker.py` with:

1. `get_dismiss_signal(db: AsyncSession, tenant_id: UUID, sender_email: str, days: int = 30, threshold: int = 3) -> str` — returns a dismiss signal string for injection into the scoring prompt, or empty string if below threshold:
   - Run a COUNT query: `SELECT COUNT(*) FROM email_drafts ed JOIN emails e ON ed.email_id = e.id WHERE e.tenant_id = :tid AND e.sender_email = :sender AND ed.status = 'dismissed' AND ed.updated_at >= now() - make_interval(days => :days)`. Use `sa_text` with `.bindparams()` — same pattern as `_check_daily_scoring_cap` in gmail_sync.py. IMPORTANT: Use `make_interval(days => :days)` not string interpolation for the interval (avoids SQL injection and parameterization issues).
   - If count >= threshold: return `f"\nDISMISS SIGNAL: User has dismissed {count} draft(s) for this sender in the past {days} days. Score DOWN: this sender category produces drafts the user doesn't want to send."`.
   - If count < threshold: return `""`.
   - Wrap in try/except — return empty string on any failure (non-fatal, same as scoring pattern).

Add to `backend/src/flywheel/config.py` Settings class:
- `voice_update_min_edits: int = 1` — minimum edit count before voice update triggers (1 = every edit)
- `dismiss_lookback_days: int = 30` — rolling window for dismiss signal
- `dismiss_threshold: int = 3` — minimum dismissals to trigger scoring signal
