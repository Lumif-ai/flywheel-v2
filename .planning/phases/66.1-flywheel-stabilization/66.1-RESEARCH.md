# Phase 66.1: Flywheel Stabilization - Research

**Researched:** 2026-03-29
**Domain:** Bug fixes, migration repair, engine hardening (Python/SQLAlchemy/Alembic/Supabase)

## Summary

Phase 66.1 is a stabilization phase -- no new features, only fixing 18 discovered issues that prevent the flywheel engine from running end-to-end. The issues span three layers: (1) infrastructure (env var mismatch blocking Supabase Storage, migration 034 FK failure, missing `updated_at` trigger), (2) engine correctness (timezone bug, title-matching false positives, None guards, stale model constant, N+1 query), and (3) robustness (NULL events_log guard, execution caps, race conditions, private import cleanup, MCP timeout).

All issues are in existing code at known file locations. The fixes are surgical -- most are 1-10 line changes. The risk is not complexity but rather regression: changes to `_append_event_atomic`, migration rewrites, and env var consolidation touch shared infrastructure used by all engines.

**Primary recommendation:** Fix in dependency order: infrastructure first (env vars, migrations), then correctness (timezone, guards), then robustness (caps, race conditions). Each fix should be independently testable.

## Architecture Patterns

### Pattern 1: Env Var Access via Settings Singleton
**What:** All configuration should flow through `flywheel.config.settings`, not raw `os.environ` lookups.
**Current state (BUG):** Three files use `os.environ["SUPABASE_URL"]` and `os.environ["SUPABASE_SERVICE_ROLE_KEY"]` directly:
- `services/document_storage.py` (lines 35-36, 58-59)
- `engines/meeting_processor_web.py` (lines 367-368)
- `api/relationships.py` (lines 715-716)

The Pydantic Settings field is `supabase_service_key` (maps to env var `SUPABASE_SERVICE_KEY`), but the raw lookups expect `SUPABASE_SERVICE_ROLE_KEY`. This mismatch means: either the env var name in deployment is `SUPABASE_SERVICE_ROLE_KEY` (and Settings never picks it up), or the env var is `SUPABASE_SERVICE_KEY` (and the raw lookups fail with KeyError).

**Fix pattern:**
```python
# BEFORE (broken -- env var name mismatch)
supabase_url = os.environ["SUPABASE_URL"]
service_key = os.environ["SUPABASE_SERVICE_ROLE_KEY"]

# AFTER (correct -- single source of truth)
from flywheel.config import settings
supabase_url = settings.supabase_url
service_key = settings.supabase_service_key
```

**Decision needed:** If the actual env var in deployment is `SUPABASE_SERVICE_ROLE_KEY`, add a Field alias to Settings:
```python
supabase_service_key: str = Field("", alias="SUPABASE_SERVICE_ROLE_KEY")
```
Or rename to `supabase_service_role_key` for clarity. Check `.env` / Railway config to determine the actual env var name.
**Confidence:** HIGH -- verified by grep of all os.environ references in backend.

### Pattern 2: Alembic Migration FK Constraints
**What:** Migration 034 (tasks table) declares `ForeignKeyConstraint(["user_id"], ["profiles.id"])` but the profiles table is managed by Supabase Auth. Not all user UUIDs in the application may exist in the profiles table.
**Evidence:** Migration 032 (meetings table) explicitly states in a comment: "No FK on user_id -- profiles table is managed by Supabase Auth". This was a deliberate decision for meetings that 034 contradicts.
**Fix:** Remove the FK on `user_id -> profiles.id` from migration 034. Keep the FK on `tenant_id -> tenants.id` (tenants are always present). Add `updated_at` trigger (missing from 034 but present in all other tables with `updated_at` columns).
**Confidence:** HIGH -- verified by reading both migration files.

### Pattern 3: Timezone-Aware Date Queries
**What:** Stage 3 queries "today's meetings" using `date.today()` which returns server-local date. On Railway (UTC), this is wrong for US users after 5pm PT (next day in UTC).
**Fix pattern:**
```python
# BEFORE (server timezone -- wrong)
today = date.today()

# AFTER (UTC-aware, or user-timezone-aware)
from datetime import datetime, timezone
today_utc = datetime.now(timezone.utc).date()
# Better: use user's timezone if stored, fallback to UTC
```
**Note:** The brief header (`_compose_daily_brief` line 612) also uses `date.today()` -- same fix needed there.
**Confidence:** HIGH -- verified at lines 269 and 612.

### Pattern 4: Haiku Model Constant
**What:** The codebase standardizes on `_HAIKU_MODEL = "claude-haiku-4-5-20251001"` (defined in 7+ files), but `flywheel_ritual.py` line 484 hardcodes the older `"claude-haiku-4-20250514"`.
**Fix:** Define `_HAIKU_MODEL` constant at module level in `flywheel_ritual.py` matching the rest of the codebase, or import from a shared location.
**Confidence:** HIGH -- verified by grep across all engines.

### Pattern 5: N+1 Query in _filter_unprepped
**What:** `_filter_unprepped()` (line 887-918) opens a new DB session per meeting to check if it has a completed prep run. For 10 meetings, that is 10 separate DB round-trips.
**Fix pattern:**
```python
# BEFORE (N+1 -- one query per meeting)
for meeting in meetings:
    async with factory() as session:
        existing_prep = await session.execute(...)

# AFTER (batch query -- one query for all)
async with factory() as session:
    await _set_rls_context(session, tenant_id, None)
    titles = [m.title for m in meetings if m.title]
    if titles:
        result = await session.execute(
            select(SkillRun.input_text).where(
                SkillRun.tenant_id == tenant_id,
                SkillRun.skill_name == "meeting-prep",
                SkillRun.status == "completed",
            )
        )
        prepped_inputs = {row[0] for row in result.all()}
    # Then filter in Python
    for meeting in meetings:
        if not meeting.title or not any(meeting.title in inp for inp in prepped_inputs):
            unprepped.append(meeting)
```
**Confidence:** HIGH -- verified the N+1 pattern in the source code.

### Anti-Patterns to Avoid
- **Importing private functions across modules:** `flywheel_ritual.py` imports `_append_event_atomic`, `_execute_meeting_prep`, `_execute_meeting_processor`, `_execute_account_meeting_prep`, `_execute_with_tools`, `_load_skill_from_db` from `skill_executor.py`. These are all underscore-prefixed (private). While functional, this creates tight coupling. The stabilization should document this debt but NOT refactor it now (too risky for a stabilization phase).
- **Silent NULL concatenation:** PostgreSQL `NULL || jsonb_array = NULL`. The `_append_event_atomic` function does `events_log || CAST(:event AS jsonb)` without COALESCE, meaning if events_log is ever NULL, all events are silently lost.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Env var access | Direct `os.environ` lookups | `flywheel.config.settings` | Single source of truth, type-safe, .env file support |
| HTML escaping | Custom regex/replace | `html.escape()` from stdlib | The existing `_escape()` at line 855 is adequate but could use stdlib |
| Timezone conversion | Manual UTC offset math | `datetime.now(timezone.utc)` or `zoneinfo` | Avoid off-by-one day bugs |
| JSON serialization guards | Manual None checks | `json.dumps(x, default=str)` | Handles UUID, datetime, etc. |

**Key insight:** This phase is about fixing existing code, not building new abstractions. Every fix should be as minimal as possible to avoid new bugs.

## Common Pitfalls

### Pitfall 1: Migration Rewrite Breaking Existing Databases
**What goes wrong:** Rewriting migration 034 after it has already been applied to a database means `alembic upgrade head` skips it (already at head). The FK constraint remains.
**Why it happens:** Alembic tracks revision IDs, not file contents.
**How to avoid:** Create a new migration 035 that drops the FK constraint and adds the trigger. Do NOT modify 034 in-place unless no database has ever run it.
**Warning signs:** `alembic upgrade head` reports "no new migrations" but FK still exists.

### Pitfall 2: _append_event_atomic NULL Guard Breaking Existing Behavior
**What goes wrong:** Adding COALESCE changes the SQL. If any other code depends on the exact NULL behavior (unlikely but possible), it could break.
**How to avoid:** The fix is safe: `COALESCE(events_log, '[]'::jsonb) || CAST(:event AS jsonb)`. This only changes behavior when events_log IS NULL, which is a bug state anyway.
**Warning signs:** Events not appearing in the SSE stream.

### Pitfall 3: Title Matching False Positives in _filter_unprepped
**What goes wrong:** `SkillRun.input_text.contains(meeting.title)` is a substring match. A meeting titled "Q1" would match any prep run whose input_text contains "Q1" anywhere (e.g., "Q1 Pipeline Review with Acme Corp").
**Why it happens:** `contains()` maps to SQL `LIKE '%Q1%'` which is overly broad.
**How to avoid:** Use a more specific match. Options: (a) store meeting_id on SkillRun, (b) use `input_text.startswith()` or exact match on a structured field, (c) check meeting_id in skill_context metadata.
**Warning signs:** Meetings appear "already prepped" when they have not been prepped.

### Pitfall 4: Stage 4 Crashing on Missing Tasks Table
**What goes wrong:** If migration 034 failed, the tasks table does not exist. Stage 4 queries `select(Task).where(...)` which raises `ProgrammingError: relation "tasks" does not exist`.
**Why it happens:** The engine assumes all migrations are applied.
**How to avoid:** Wrap Stage 4 task query in a try/except that catches `ProgrammingError` and gracefully returns 0 tasks.
**Warning signs:** Entire flywheel run fails at Stage 4 even though Stages 1-3 succeed.

### Pitfall 5: user_id is None in Stage 4 Query
**What goes wrong:** `Task.user_id == user_id` where `user_id` is `None` generates `user_id IS NULL` in SQL, which matches no tasks (user_id is NOT NULL in the schema).
**Why it happens:** The engine signature allows `user_id: UUID | None` but Stage 4 uses it in a WHERE clause without guarding.
**How to avoid:** Guard: `if user_id is None: return early with 0 tasks`.
**Warning signs:** Stage 4 always reports "0 tasks" even when confirmed tasks exist.

### Pitfall 6: Meeting Processing Without Caps
**What goes wrong:** Stage 2 processes ALL unprocessed meetings with no limit. If 100 meetings are pending, it runs 100 LLM calls sequentially, taking 30+ minutes.
**Why it happens:** The spec says "no caps" for processing, but practical runtime limits matter.
**How to avoid:** Add a configurable cap (e.g., `MAX_MEETINGS_TO_PROCESS = 20`) with an event message noting remaining meetings.
**Warning signs:** MCP timeout (5 min) exceeded; flywheel run appears to hang.

### Fix 4: Batch _filter_unprepped Query
```python
# Source: backend/src/flywheel/engines/flywheel_ritual.py lines 887-918
async def _filter_unprepped(
    factory: async_sessionmaker,
    tenant_id: UUID,
    meetings: list,
) -> list:
    if not meetings:
        return []

    titled = [m for m in meetings if m.title]
    untitled = [m for m in meetings if not m.title]

    if not titled:
        return untitled  # All untitled = all unprepped

    async with factory() as session:
        await _set_rls_context(session, tenant_id, None)
        result = await session.execute(
            select(SkillRun.input_text).where(
                SkillRun.tenant_id == tenant_id,
                SkillRun.skill_name == "meeting-prep",
                SkillRun.status == "completed",
            )
        )
        prepped_texts = [row[0] or "" for row in result.all()]

    unprepped = list(untitled)  # Untitled always unprepped
    for meeting in titled:
        if not any(meeting.title in pt for pt in prepped_texts):
            unprepped.append(meeting)

    return unprepped
```

### Fix 5: Stage 4 Guards

```python
# Guard against None user_id and missing tasks table
async def _stage_4_execute(...):
    if user_id is None:
        logger.warning("Stage 4 skipped: user_id is None")
        await _append_event_atomic(factory, run_id, {
            "event": "stage",
            "data": {"stage": "executing", "message": "Task execution skipped (no user context)."},
        })
        return

    try:
        async with factory() as session:
            await _set_rls_context(session, tenant_id, user_id)
            result = await session.execute(
                select(Task).where(...)
            )
            executable_tasks = result.scalars().all()
    except Exception as e:
        # Catches ProgrammingError if tasks table doesn't exist
        logger.warning("Stage 4: could not query tasks: %s", e)
        await _append_event_atomic(factory, run_id, {
            "event": "stage",
            "data": {"stage": "executing", "message": f"Task execution skipped: {e}"},
        })
        return
```

   - What we know: Current `contains()` is too loose; NULL titles are handled
   - What's unclear: Is there a better signal than input_text substring matching?
   - Recommendation: For now, tighten the match. Long-term, store meeting_id on SkillRun or in skill_context.

3. **MCP timeout for flywheel ritual**
   - What we know: MCP polls for ~5 minutes (30 intervals). Flywheel with 20 meetings could take 10+ minutes.
   - What's unclear: Can the MCP timeout be increased without Claude Code timing out?
   - Recommendation: Increase polling cap to ~10 minutes. Also add execution caps to keep runtime under 5 minutes for typical loads.

4. **Migration 034 -- rewrite vs. follow-up migration**
   - What we know: If 034 has never been applied to production, rewriting it is safe
   - What's unclear: Has any database run 034?
   - Recommendation: If production has run 034, create 035 to drop FK + add trigger. If not, rewrite 034 in place.
