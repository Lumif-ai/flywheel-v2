Add 10 new methods to the FlywheelClient class in the "Public API" section, after the existing methods. Follow the EXACT existing pattern: thin methods returning dict via self._request(), docstrings in "HTTP_METHOD /api/v1/path -- description" format, no business logic.

The 10 methods and their signatures:

1. fetch_skills() -> dict
   GET /api/v1/skills/
   No params needed. Returns {"items": [...]}.

2. fetch_skill_prompt(skill_name: str) -> dict
   GET /api/v1/skills/{skill_name}/prompt
   Path parameter only.

3. fetch_meetings(time: str | None = None, limit: int = 50, offset: int = 0) -> dict
   GET /api/v1/meetings/
   params={"limit": limit, "offset": offset}, conditionally add "time" if not None.

4. fetch_upcoming(limit: int = 10) -> dict
   GET /api/v1/meetings/
   params={"time": "upcoming", "limit": limit}
   This is a convenience wrapper (not a separate endpoint).

5. fetch_tasks(status: str | None = None, limit: int = 50) -> dict
   GET /api/v1/tasks/
   params={"limit": limit}, conditionally add "status" if not None.

6. fetch_account(account_id: str) -> dict
   GET /api/v1/accounts/{account_id}
   Path parameter only.

7. sync_meetings() -> dict
   POST /api/v1/meetings/sync
   No body needed.

8. save_document(title: str, skill_name: str, markdown_content: str, metadata: dict | None = None) -> dict
   POST /api/v1/documents/from-content
   json={"title": title, "skill_name": skill_name, "markdown_content": markdown_content, "metadata": metadata or {}}

9. save_meeting_summary(meeting_id: str, ai_summary: str, processing_status: str = "completed") -> dict
   PATCH /api/v1/meetings/{meeting_id}
   json={"ai_summary": ai_summary, "processing_status": processing_status}

10. update_task(task_id: str, **fields) -> dict
    PATCH /api/v1/tasks/{task_id}
    json=fields (pass through all keyword arguments as the request body)

Key rules:
- GET methods use params= kwarg, POST/PATCH use json= kwarg
- Optional params: build dict conditionally (only include key if value is not None)
- All methods return dict (from _request -> resp.json())
- No try/except in methods -- _request() handles all error cases
