**Gap 3 fix -- Clear concurrent limit UX:**

The backend already returns a structured 429 response with `{"error": "ConcurrentRunLimitExceeded", "message": "Maximum 3 concurrent runs. Wait for a run to complete."}` in the detail field. The frontend catch block (lines 147-157 of store.ts) currently shows the raw error message as a generic error.

1. In `frontend/src/features/chat/store.ts`, update the catch block in `sendMessage`:
   - Check if the error is an API error with status 429 and detail containing "ConcurrentRunLimitExceeded"
   - The `api` helper likely wraps errors -- check how `api.post` throws errors. The error message may already contain the detail message. Look at `frontend/src/lib/api.ts` to see how HTTP errors are surfaced.
   - If the error message contains "Maximum 3 concurrent runs" or "ConcurrentRunLimitExceeded", create the assistant message with a distinct status. Use `status: 'complete'` (not 'error') so it renders as a normal message, not an error box. Set content to a user-friendly message: "You have 3 skills running right now. Please wait for one to finish before starting another."
   - If it's any other error, keep the existing generic error handling.

2. The key change in the catch block:
   ```typescript
   } catch (err) {
     const message = err instanceof Error ? err.message : 'Failed to process message'
     const isConcurrentLimit = message.includes('concurrent') || message.includes('ConcurrentRunLimitExceeded')

     const errorMsg: ChatMessage = {
       id: crypto.randomUUID(),
       role: 'assistant',
       content: isConcurrentLimit
         ? 'You have 3 skills running right now. Please wait for one to finish before starting another.'
         : message,
       timestamp: new Date(),
       status: isConcurrentLimit ? 'complete' : 'error',
     }
     addMessage(errorMsg)
   }
   ```

This approach accepts the "reject with clear message" simplification rather than implementing a full deferred queue, which is appropriate for the current stage.
