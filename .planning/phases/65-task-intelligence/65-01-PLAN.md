2. Add a NEW async function `_compute_task_signals(db, tenant_id, user_id, now) -> dict` that:
   - Executes _TASK_SIGNALS_SQL with params {"tid": tenant_id, "uid": user_id, "now": now}
   - Returns a dict with keys: tasks_detected, tasks_in_review, tasks_overdue (all int, defaulting to 0 if no rows)

3. Add a NEW separate task signals cache `_task_signals_cache: dict[str, tuple[float, dict]] = {}` keyed by `f"{tenant_id}:{user_id}"`. Do NOT modify the existing `_signals_cache` — that cache is tenant-scoped for relationship signals and changing its key would give each user a cold cache on every request. Use the same TTL pattern as the existing cache (check what TTL _signals_cache uses and match it). Add `_get_task_cached(key)` and `_set_task_cached(key, value)` helper functions mirroring the existing `_get_cached`/`_set_cached` pattern but operating on `_task_signals_cache`.

4. Restructure `get_signals()` so task signals run on BOTH cache-hit and cache-miss paths. CRITICAL: The existing code has an early return at ~lines 197-199 (`cached = _get_cached(...); if cached is not None: return cached`) that returns the cached `SignalsResponse` before any new code would run. You MUST remove that early return and restructure into a two-layer approach:

   **Step A — Relationship signals (tenant-scoped, existing cache):**
   - Check `_signals_cache` via `_get_cached(user.tenant_id)`
   - If cache miss: compute via `_compute_all_signals()`, build `SignalsResponse`, call `_set_cached()`
   - Either way, assign the result to a local `response` variable — do NOT return yet

   **Step B — Task signals (user-scoped, new cache, runs ALWAYS):**
   - Build key `f"{user.tenant_id}:{user.sub}"`
   - Check `_task_signals_cache` via `_get_task_cached(key)`
   - If cache miss or expired: call `_compute_task_signals(db, user.tenant_id, user.sub, now)`
   - Store in `_task_signals_cache` via `_set_task_cached(key, result)`
   - Note: `now` must be computed before Step B even if Step A was a cache hit

   **Step C — Merge and return:**
   - Set `response.tasks_detected`, `response.tasks_in_review`, `response.tasks_overdue` from the task signals dict
   - Return the merged response

   The refactored function shape:
   ```python
   async def get_signals(db, user) -> SignalsResponse:
       now = datetime.datetime.now(datetime.timezone.utc)

       # Step A: relationship signals (tenant-scoped cache)
       response = _get_cached(user.tenant_id)
       if response is None:
           all_counts = await _compute_all_signals(db, user.tenant_id, now)
           # ... build type_badges, grand_total ...
           response = SignalsResponse(types=type_badges, total=grand_total)
           _set_cached(user.tenant_id, response)

       # Step B: task signals (user-scoped cache — always runs)
       task_key = f"{user.tenant_id}:{user.sub}"
       task_counts = _get_task_cached(task_key)
       if task_counts is None:
           task_counts = await _compute_task_signals(db, user.tenant_id, user.sub, now)
           _set_task_cached(task_key, task_counts)

       # Step C: merge task counts into response
       response.tasks_detected = task_counts["tasks_detected"]
       response.tasks_in_review = task_counts["tasks_in_review"]
       response.tasks_overdue = task_counts["tasks_overdue"]
       return response
   ```

5. Extend the SignalsResponse Pydantic model with three new optional fields:
   - tasks_detected: int = 0
   - tasks_in_review: int = 0
   - tasks_overdue: int = 0
   These default to 0 so existing clients that don't read them are unaffected.
