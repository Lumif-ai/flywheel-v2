**2. Create `backend/src/flywheel/engines/email_scorer.py`**

Module structure:

```
Imports: anthropic (AsyncAnthropic), json, re, logging, datetime, UUID
         SQLAlchemy: select, text as sa_text, or_
         pg_insert from sqlalchemy.dialects.postgresql
         Models: Email, EmailScore, ContextEntity, ContextEntry
         Settings: flywheel.config.settings
         Session: flywheel.db.session.tenant_session, get_session_factory
```

Constants:
- `_HAIKU_MODEL = "claude-haiku-4-5-20251001"` (same as gmail_sync.py)
- `SCORE_SYSTEM_PROMPT` -- full scoring prompt (details below)

Functions (all async):

**`_lookup_sender_entity(db, tenant_id, sender_email) -> ContextEntity | None`**
- Extract domain from sender_email (split on @)
- Query context_entities WHERE tenant_id matches AND (name ILIKE %domain% OR aliases @> [sender_email])
- Limit 1, return scalar_one_or_none()
- Follow the exact query pattern from research (Pattern 4)

**`_search_context_entries(db, tenant_id, subject) -> list[ContextEntry]`**
- Strip Re:/Fwd:/Fw: prefixes from subject using regex
- If cleaned subject is empty, return []
- Query context_entries WHERE tenant_id matches AND deleted_at IS NULL AND search_vector @@ plainto_tsquery('english', cleaned_subject)
- Order by ts_rank DESC, limit 3
- Use sa_text() for the FTS WHERE clause and ORDER BY (not ORM -- FTS requires raw SQL fragments)
- Follow the exact query pattern from research (Pattern 5)

**`_build_score_prompt(email, sender_entity, context_entries) -> tuple[str, str]`**
Returns (system_prompt, user_message).

System prompt (`SCORE_SYSTEM_PROMPT` constant) must include:
1. Role: "You are an email priority scorer for a busy professional"
2. 1-5 scale with definitions (5=CRITICAL same-day, 4=IMPORTANT 24h, 3=NORMAL, 2=LOW FYI, 1=NOISE marketing)
3. SCORING BIAS section: "When uncertain, score UP. A false positive is acceptable. A false negative is a critical product failure." (SCORE-09)
4. Categories: meeting_followup, deal_related, action_required, informational, marketing, personal
5. Suggested actions: notify, draft_reply, file, archive
6. Signal hierarchy table as scoring guidance (from research):
   - Known entity mention_count >= 5: score higher
   - Entity type "company" with matching context entries: score higher
   - Subject matches active context entry: score higher
   - Label IMPORTANT: boost
   - Label CATEGORY_PROMOTIONS/CATEGORY_UPDATES: baseline low
   - Urgency words (urgent, deadline, ASAP, closing, by [date]): boost
   - is_replied=False and received > 7 days ago: stale thread boost
7. Three few-shot examples anchoring scores 5, 4, and 2 (hardcoded realistic samples, no tenant data):
   - Score 5: Known entity CEO + "term sheet review" subject + matching deal context entry + urgency
   - Score 4: Known entity + "follow up on proposal" + no urgency keyword
   - Score 2: Unknown sender + CATEGORY_PROMOTIONS label + generic marketing subject
8. JSON output format specification (priority, category, suggested_action, reasoning, context_refs)
9. Instruction: "Return ONLY a JSON object" -- same constraint as voice profile extraction

User message: structured block with:
- CONTEXT AVAILABLE section: sender_entity JSON (or "No known entity") + matching context_entries as array of {id, file_name, content_snippet (first 200 chars of detail)}
- EMAIL TO SCORE section: From, Subject, Snippet, Labels, is_replied, received_at
- Pass entity IDs and entry IDs in the context block so the LLM can reference them

**`_parse_score_response(text, valid_entity_ids, valid_entry_ids) -> dict`**
- json.loads() with re.search(r"\{.*\}", text, re.DOTALL) fallback (same pattern as gmail_sync.py voice profile)
- Validate priority is int 1-5, clamp if out of range
- Validate category is one of the 6 allowed values, default to "informational" if invalid
- Validate suggested_action is one of the 4 allowed values, default to "file" if invalid
- Validate context_refs: filter out any ref whose id is NOT in valid_entity_ids or valid_entry_ids (prevent hallucinated references)
- Return cleaned dict

**`_upsert_email_score(db, tenant_id, email_id, score_dict, sender_entity_id) -> None`**
- Use pg_insert(EmailScore).values(...).on_conflict_do_update(constraint="uq_email_score_email", set_={...})
- Set all fields including scored_at = now()
- Do NOT commit -- caller commits (caller-commits pattern from Phase 2)

**`async def score_email(db, tenant_id, email, api_key=None) -> dict | None`**
Main entry point. Orchestrates the full scoring pipeline for a single email:
1. `_lookup_sender_entity(db, tenant_id, email.sender_email)`
2. `_search_context_entries(db, tenant_id, email.subject)`
3. Build prompt via `_build_score_prompt(email, entity, entries)`
4. Call AsyncAnthropic with api_key (or settings.flywheel_subsidy_api_key if None) using _HAIKU_MODEL, max_tokens=500
5. Parse response via `_parse_score_response()`
6. `_upsert_email_score(db, tenant_id, email.id, score_dict, entity.id if entity else None)`
7. Return the parsed score dict (for caller logging/chaining)
8. On any exception: log error with email.id (no PII -- no subject/sender in logs), return None (non-fatal, same pattern as voice_profile_init)

**Important implementation notes:**
- Use `settings.flywheel_subsidy_api_key` as the default api_key (same as voice_profile_init)
- The caller (gmail_sync.py in Plan 02) will pass the tenant-scoped db session -- the engine does NOT create its own session
- All context_refs IDs must be validated against pre-fetched sets before storage
- No email body is accessed -- scoring uses only sender, subject, snippet, labels (architecture decision from Phase 1)
- Zero PII in log output -- only email.id, tenant_id, priority result
