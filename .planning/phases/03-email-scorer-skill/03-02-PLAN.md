**2. Add daily cap helper function:**

```python
async def _check_daily_scoring_cap(db, tenant_id: UUID, cap: int = 500) -> int:
    """Return remaining scoring budget for today. Default cap: 500/day."""
```
- Query: `SELECT COUNT(*) FROM email_scores es JOIN emails e ON es.email_id = e.id WHERE e.tenant_id = :tid AND es.scored_at >= :today`
- `:today` = start of today UTC (datetime.now(timezone.utc).replace(hour=0, minute=0, second=0, microsecond=0))
- Return max(0, cap - count)
- Use sa_text() for the join query since we need the cross-table count

**3. Add scoring integration function:**

```python
async def _score_new_emails(db, tenant_id: UUID, email_ids: list[UUID]) -> int:
    """Score a batch of newly synced emails. Returns count of emails scored."""
```
Implementation:
1. Check daily cap via `_check_daily_scoring_cap(db, tenant_id)` -- if remaining == 0, log "Daily scoring cap reached for tenant {tenant_id}" and return 0
2. Truncate email_ids to remaining cap (score at most `remaining` emails)
3. Load Email rows for the truncated list: `SELECT * FROM emails WHERE id IN (:ids)` (they're already in the session from upsert, but reload to be safe)
4. For each email in the list:
   a. Call `score_email(db, tenant_id, email)` -- this is the engine from Plan 01
   b. If score_email returns None (failure), log warning and continue (non-fatal per-email)
   c. Increment scored count
5. Commit once after all emails are scored (single commit for the batch)
6. Log "Scored {count}/{total} emails for tenant {tenant_id}"
7. Return scored count

**4. Modify `sync_gmail()` to collect new email IDs:**

Currently `sync_gmail` calls `upsert_email()` for each message but does not track which emails were newly created vs. updated. Modify to collect the IDs:

In `_full_sync` and the incremental sync path, after `upsert_email()` returns, collect the email ID. The simplest approach: `upsert_email()` already executes the pg_insert and the row exists. After the upsert, query the email by gmail_message_id to get its UUID (or modify upsert_email to return the ID via RETURNING clause on the pg_insert).

**Preferred approach:** Modify `upsert_email()` to add `.returning(Email.id)` to the pg_insert statement and return the UUID. Then collect returned IDs in a list during the sync loop.

In `sync_gmail()`, after the existing commit, call:
```python
if new_email_ids:
    try:
        scored = await _score_new_emails(db, intg.tenant_id, new_email_ids)
        logger.info("Scored %d new emails for integration %s", scored, intg.id)
    except Exception:
        logger.exception("Scoring failed for integration %s", intg.id)
        # Non-fatal -- sync is already committed, scoring failure doesn't lose emails
```

Place this AFTER the existing `await db.commit()` that commits the upserted emails. Scoring is a separate concern -- if scoring fails, the emails are already safely stored.

**5. Thread priority (SCORE-07) -- add utility function for read-time computation:**

```python
async def get_thread_priority(db, tenant_id: UUID, gmail_thread_id: str) -> int | None:
    """Compute thread priority as MAX(priority) of unhandled messages in the thread.

    Returns None if no scored messages exist for the thread.
    Used by Phase 5 API layer -- exported for import.
    """
```
SQL: `SELECT MAX(es.priority) FROM email_scores es JOIN emails e ON es.email_id = e.id WHERE e.tenant_id = :tid AND e.gmail_thread_id = :thread_id AND e.is_replied = FALSE`

This is a read-time computation, not stored. Document the SQL clearly for Phase 5 to adopt.

**6. SCORE-08 (re-score on new message):**

This is automatically handled by the design: when a new message arrives in an existing thread via incremental sync, it gets upserted as a new Email row and its ID is collected in `new_email_ids`. `_score_new_emails` then scores it. The thread priority (computed at read time) automatically reflects the new score. No special re-scoring logic needed -- the architecture handles it.

Add a comment in `_score_new_emails` noting this: "SCORE-08: New messages in existing threads are scored here automatically. Thread priority (SCORE-07) is computed at read time via get_thread_priority()."

**Important implementation notes:**
- Scoring happens AFTER the email upsert commit -- scoring failure cannot lose synced emails
- Per-email exceptions are caught inside the loop -- one bad email doesn't block the rest
- Daily cap query uses scored_at >= today UTC, not created_at (re-scoring doesn't double-count if we use scored_at for the "today" window, but since we upsert on conflict, the count is accurate)
- Zero PII in logs -- only email.id, tenant_id, counts
