01 Executive Summary

The 30-second version. What Flywheel is, where it's going, and why the architecture matters.

What It Is

A compound intelligence operating system for teams. GTM is the beachhead — pipeline, briefing, email, document library — but the architecture is domain-agnostic. A shared intelligence layer accumulates structured knowledge from every meeting, email, and interaction across any business function. Claude Code + MCP is the primary interface; a React web app provides visual operating surfaces.

What Makes It Defensible

Not AI capability — that commoditizes. The moat is: deep business knowledge that learns continuously from every interaction across every module. Connected agents — sales knows what legal flagged, product knows what customers asked for in calls, HR knows which roles the pipeline demands. Background compounding — intelligence accumulates whether you ask or not. The more modules, the smarter each one gets.

Where It's Going

The heavy lifting is done. The shared layer (email, meetings, contacts, documents, context store, skills, browser automation) is vertical-agnostic. Adding a new module — HR, Product Management, Insurance Brokerage, Legal, Finance — is domain skills + domain tables + a domain page. No new infrastructure. The module-per-tenant architecture is production-ready.

Architecture Thesis

Every module shares the same shared layer (email, meetings, contacts, documents, context store, browser automation) and the same platform layer (auth, skill engine, MCP, database). Only the module layer differs. One codebase, one deployment, one data model — with per-tenant module configuration via a single JSONB field. Within a single organization, a recruiting team's candidate interviews enrich the same context store that the sales team's discovery calls feed. Cross-module intelligence compounds automatically — no integrations to configure between departments.

Any Domain, Same Pattern

The platform can automate workflows in any domain. Each module reuses the shared layer (email, meetings, contacts, documents, context store, browser automation, skill engine) and adds only domain-specific skills, tables, and a page. Because 70-80% of the infrastructure is already built, new modules go from concept to working MVP in weeks, not months.

Speed to Market: Weeks, Not Months

A traditional vertical SaaS company rebuilds email integration, auth, document management, and AI pipelines for every new product. We build it once. Each new module inherits everything listed in the shared layer — the module author writes only domain skills and domain tables.

What this means practically: The entire GTM operating system — email sync, meeting intelligence, pipeline management, contact enrichment, document generation, browser automation — was built as domain-agnostic infrastructure. A new vertical module can go from concept to customer-facing MVP in weeks because the hard problems (auth, multi-tenancy, email integration, AI execution, PDF/DOCX export, browser automation) are already solved. The infrastructure that took us 13 milestones to build and harden is reused wholesale for every new vertical.

Why We Can Move This Fast

MCP + Skills as the real multiplier: The platform exposes 30+ tools via the open MCP protocol — a standard supported by Anthropic, OpenAI, Google, and the broader AI ecosystem. Any MCP-compatible client (Claude Code, Cursor, Windsurf, custom agents) can read the context store, run skills, manage pipeline data, and trigger workflows. We built on Claude Code, but the architecture is client-agnostic. If a better AI development environment emerges tomorrow, the MCP layer means zero migration cost.

Skills as deployable domain knowledge: Adding a new vertical doesn't require code releases. Domain expertise is captured in SKILL.md files — declarative prompts with structured output schemas. A domain expert (insurance underwriter, construction PM, lease analyst) can define a workflow as a skill and deploy it via flywheel db seed. No backend changes, no frontend changes, no deployment pipeline.

Multi-tenancy as scale lever: RLS-based tenant isolation means onboarding a new customer is creating a row, not spinning up infrastructure. One deployment serves every tenant across every vertical. A construction compliance firm and an insurance brokerage run on the same codebase, same database, same deployment — differentiated only by their tenant.settings.modules configuration.

Skill Marketplace Opportunity

The SKILL.md format is fully declarative — a domain expert can define a workflow without touching backend code. The architecture already supports: per-tenant skill enablement, structured output schemas, context store integration, and document export. This opens a natural path to a third-party skill marketplace where domain experts (insurance underwriters, legal analysts, recruiting specialists) build and publish skills that run on the platform. Revenue model: platform fee on paid skills. The compound context store is the ecosystem lock-in — skills produce better output on Flywheel than standalone because they operate on months of accumulated business intelligence, not a blank prompt.

One Tenant, One Data Model

Tenants are fully RLS-isolated — no data crosses between companies. Within a single organization, all modules share the same contacts, companies, meetings, and documents. No duplicate data entry across workflows. A contact added during a sales call is immediately available in a compliance review. A meeting transcript processed for deal context is the same transcript available for legal clause extraction. The shared data model eliminates the integration tax that plagues multi-tool stacks.

07 Key Flows

Sequence diagrams for the three most critical data flows in the system.

Flow 1: Skill Execution (e.g., One-Pager)

Two Entry Points — Same Backend

Primary (Claude Code): User types a command in Claude Code → Claude calls MCP tool flywheel_run_skill → MCP server proxies to POST /skills/runs → polls for completion → returns output to Claude → Claude presents results in terminal.
Secondary (Web UI): User clicks "Run" in browser → POST /skills/runs → SSE stream via EventSource → live progress in UI.

1
Claude Code / MCP
User invokes skill via Claude Code (e.g., /meeting-prep or natural language). Claude calls flywheel_run_skill(name, input) MCP tool. The MCP server (cli/flywheel_mcp/server.py) proxies this as POST /skills/runs to the backend API over HTTP.
2
API Gateway
Validate tenant access, check rate limits (per-tenant concurrent run limit), create SkillRun record (status: pending). Return {run_id, status}. Web UI subscribes to SSE stream; MCP server polls GET /skills/runs/{id} until complete.
3
Skill Executor
execute_run() → load skill prompt from DB, build messages array, inject system prompt. Protected skills get an anti-extraction prefix prepended at runtime (never stored in DB) to prevent prompt copying. If output_schema exists, set output_config.format.json_schema (API-level enforcement). Use token_budget from skill metadata (or 4096 fallback).
4
Execution Gateway
Anthropic API call with tool_use loop. Circuit breaker wraps all calls (3 failures → 60s OPEN → HALF-OPEN probe). Each tool result emitted as SSE event. Skills can call MCP tools (context store reads/writes, web search) during execution.
5
Output Renderer
Detect structured JSON (schema_version check), select Jinja2 template, render HTML. Non-fatal: rendering failure doesn't fail the run.
6
Database + Response
Save SkillRun.output + rendered_html. Create Document in library. MCP path: MCP server returns output text to Claude, which presents it in the terminal. Web path: Emit final SSE done event; React renders via SkillRenderer → OnePagerRenderer (or GenericRenderer).

Flow 2: Email Intelligence Pipeline

1
Gmail API
Incremental sync via historyId. Per-integration 60s timeout. asyncio.gather(return_exceptions=True) for concurrent integrations. ON CONFLICT DO UPDATE dedup at DB layer. Can also be triggered via MCP tool flywheel_sync_meetings.
2
Email Scorer
Five-tier priority scoring using sender history, topic relevance, relationship context. Confidence-based routing: high-confidence auto-files, uncertain goes to review queue.
3
Email Drafter
AI-generated reply using EmailVoiceProfile (learned from 50+ samples). Draft stored as EmailDraft linked to original email.
4
Context Extractor
Extract contacts, company mentions, pain points, competitive signals. Write to ContextEntry with provenance. Queryable via MCP tools flywheel_read_context / flywheel_write_context.

Pipeline Isolation

Each stage is committed before the next begins. A failure in context extraction cannot lose email scoring results. Per-email exceptions are caught non-fatally — one bad email never blocks the batch.

Flow 3: Meeting Intelligence

1
Granola via MCP
Triggered via flywheel_sync_meetings MCP tool from Claude Code, or automatically on calendar sync. Paginated transcript fetch (page_size=30). Fuzzy title dedup against existing meetings.
2
Meeting Processor
7-stage pipeline: classify meeting type → extract 9 insight types (pain points, budget signals, competitive mentions, decisions, next steps, stakeholders, objections, timeline, relationship dynamics).
3
Context Store Writer
Entity extraction → normalization → relationship mapping → confidence scoring. Writes structured ContextEntry records linked to account. Accessible via flywheel_read_context.
4
Task Extractor
Auto-extract action items as Task records. Visible via flywheel_fetch_tasks MCP tool and web briefing page. Non-fatal: failure logs warning, doesn't fail processing.

MCP as the Primary Interface

Claude Code is the primary user interface. The MCP server exposes 30+ tools that map to every backend capability. The web UI is the secondary interface for visual surfaces (pipeline grid, briefing, email queue, document library) that benefit from persistent, interactive display. Both interfaces hit the same backend API — there is no separate "MCP backend."

Flow 4: GTM Outreach with Browser Automation

1
Claude Code
User asks Claude to "send outreach to the top 5 leads" or "scrape contacts from this directory." Claude calls Flywheel MCP tools to load pipeline data (flywheel_list_pipeline, flywheel_read_context) and reads accumulated context for personalization.
2
Outreach Drafter
Generates personalized email/LinkedIn messages using context store intelligence (pain points from meetings, competitive signals, relationship history). Messages scored against a quality bar before sending.
3
Playwright MCP
Browser automation via Playwright MCP server. Navigates to carrier portals, LinkedIn, or target websites. Fills forms, extracts structured data, takes screenshots. Per-carrier/per-portal configuration for different submission flows.
4
Pipeline Service
Records outreach as OutreachActivity linked to pipeline entry. Creates LeadMessage for tracking. Updates pipeline stage based on activity signals. All visible in pipeline grid and briefing.

Playwright is the Expansion Lever

Browser automation makes Flywheel's skill system capable of interacting with any web application — carrier portals (insurance), property listing sites (CRE), government filing systems (legal). The same Playwright MCP infrastructure that powers GTM web scraping and outreach becomes the integration layer for vertical-specific workflows without building custom API integrations.

13 Failure Modes & Resilience

How the system behaves when things go wrong. Every failure mode has been encountered in production.

Circuit Breaker (Anthropic API)

CLOSED ──(3 consecutive failures)──→ OPEN ──(60s cooldown)──→ HALF-OPEN ↑ │ │ │ │ all calls blocked │ allow 1 probe └──────────(success)───────────────────←┘ │ └──────────(probe succeeds)────────────────────────────────────────←┘ Thread-safe via threading.Lock. Process-level singleton (anthropic_breaker). Shared by chat orchestrator and skill executor.

Failure Matrix

FailureDetectionResponseData Loss?
Anthropic API down Circuit breaker trips after 3 failures All LLM calls blocked for 60s. Skills return error status. Email scoring deferred. No — emails/meetings saved, AI processing retried later
Gmail OAuth token expired TokenRevokedException Integration marked as disconnected. User prompted to re-auth. Other integrations unaffected. No — sync resumes from last historyId after re-auth
Gmail rate limit / HTTP error HttpError 404 or rate limit response One retry with history token reset. Other HTTP errors re-raised. Per-integration 60s timeout. No — incremental sync picks up from last position
Malformed LLM JSON output json.loads() failure Regex fallback extraction. If still unparseable, ValueError raised. SkillRun marked failed. No — raw output preserved in SkillRun record
Skill rendering failure Exception in output_renderer Non-fatal: warning logged, raw output available. Document still created with raw content. No — graceful degradation to raw display
Meeting processing failure 7-stage pipeline, each stage guarded Meeting marked processing_status="failed". Even status-update failure is guarded. No — transcript preserved, reprocessing possible
PgBouncer DDL rollback Migration appears to succeed but schema unchanged Manual workaround: each DDL as individual commit + alembic stamp No — but silent failure if not caught
Bad email in sync batch Per-email try/except Exception logged, email skipped. Rest of batch continues normally. Single email lost — retried on next sync cycle
Job queue worker crash Outer while True with catch-all Log exception, sleep 5s, retry. attempts < max_attempts gate prevents infinite retry. No — unclaimed jobs stay in queue for next claim

Design Principle: Stage Isolation

Multi-stage pipelines (email sync, meeting processing, skill execution) commit each stage independently. A failure in stage N cannot lose the results of stages 1 through N-1. Sub-stages (attribution enrichment, reasoning trace, task extraction) are individually try/except guarded as non-fatal.

ADR-001: Supabase PostgreSQL with RLS for multi-tenancy

Accepted
Context
Need multi-tenant isolation for a SaaS product with sensitive business data (emails, contacts, deal intel). Options ranged from app-level filtering to schema-per-tenant to database-level RLS.
Alternatives
App-level WHERE clauses: Easy to implement, easy to leak data via missed filters. Schema-per-tenant: Strong isolation, migration complexity grows linearly with tenants. Separate databases: Maximum isolation, operational overhead unmanageable for a small team.
Decision
PostgreSQL Row-Level Security via Supabase. RLS policies on every table keyed on tenant_id. Enforcement at DB level means a missing app-level filter cannot leak data.
Consequences
Positive: Data isolation enforced at the database regardless of app bugs. Supabase provides managed hosting + auth. Negative: PgBouncer connection pooling silently rolls back multi-statement DDL transactions (see ADR-006). All queries must set session tenant_id. Performance overhead of RLS policy evaluation on every query.

ADR-002: User's own Claude API key (BYOK), not platform key

Accepted
Context
AI reasoning is the core of every skill, briefing, and email draft. Who pays for the LLM tokens?
Alternatives
Platform key with usage billing: Better UX (no setup), but requires billing infrastructure, margin management, and makes us a reseller. Passthrough with markup: Complexity of proxying without the simplicity of either model.
Decision
BYOK: user provides their own Claude API key, stored AES-256 encrypted. Flywheel provides data, surfaces, and workflows — not AI capability. We are complementary to Anthropic, not competitive.
Consequences
Positive: Zero LLM cost on our side. Clear value proposition: "we build the memory, they build the brain." Better relationship with Anthropic. Negative: Setup friction (user needs Claude Code subscription). Can't control model quality or version. Token costs are user's problem to manage.

ADR-003: Generic skill executor, no hardcoded skill branches

Accepted
Context
Early versions had if skill_name == "meeting-prep" branches in the executor. Each new skill required executor code changes.
Alternatives
Keep branching: Faster for one-off skills, unmaintainable at 20+. Plugin system: Each skill as a Python module with a standard interface. More powerful, more complex.
Decision
SKILL.md declarative format with parameters JSONB. All skill-specific config (output_schema, token_budget, web_tier) flows through the parameters dict. Executor is 100% generic.
Consequences
Positive: New skills require zero executor changes. Seed pipeline handles all config. Clean separation between skill definition and execution. Negative: Some legacy hardcoded branches remain in execute_run() (tech debt, see Section 16). Less flexibility than a plugin system for skills needing custom execution logic.

ADR-004: Anthropic output_config for structured outputs

Accepted
Context
Skills like one-pager need guaranteed valid JSON matching a specific schema. Prompt-only enforcement ("output a JSON object") fails intermittently — markdown fences, missing fields, preamble text.
Alternatives
Prompt-only: Works 90% of the time, catastrophic 10%. Retry loop: Parse, validate, re-prompt on failure. Expensive, slow, still not guaranteed. Post-processing: Regex/heuristic extraction from raw output. Fragile.
Decision
Anthropic SDK output_config with json_schema format. API-level enforcement — the model physically cannot produce output that doesn't match the schema.
Consequences
Positive: 100% valid JSON. No retries. No parsing failures. Frontend can trust the shape. Negative: Tied to Anthropic SDK >=0.93.0. Schema must be maintained in sync with TypeScript interfaces. All properties must be present (nullable for optional fields).

ADR-005: PostgreSQL FOR UPDATE SKIP LOCKED as job queue

Accepted
Context
Background jobs (email sync, meeting processing) need reliable, race-free execution. Options: Redis/Celery, separate queue service, or database-native queuing.
Alternatives
Celery + Redis: Industry standard, but adds two infrastructure dependencies and operational complexity. AWS SQS / GCP Pub/Sub: Managed, but cloud-specific lock-in. Dramatiq: Lighter than Celery but still needs Redis.
Decision
PostgreSQL FOR UPDATE SKIP LOCKED in a polling loop. Zero additional infrastructure. Atomic claiming, built-in retry limits (attempts < max_attempts), transactional with the rest of the data.
Consequences
Positive: No Redis, no broker, no separate service. Jobs are ACID-consistent with business data. Easy to inspect/debug via SQL. Negative: Polling-based (5s sleep between claims). Won't scale beyond ~100 jobs/second. No priority queues. Sufficient for current scale, will need migration to proper queue at growth-stage.

ADR-006: Supabase PgBouncer DDL workaround

Accepted (workaround)
Context
Alembic migrations with multiple DDL statements silently fail on Supabase. PgBouncer commits the alembic_version update but rolls back the actual DDL. Migration appears to succeed but nothing changed.
Alternatives
Direct connection (bypass PgBouncer): Supabase doesn't expose direct connections on all plans. Single-statement migrations: One DDL per migration file. Works but creates migration file bloat.
Decision
Write migrations normally (for docs/downgrade). Apply actual DDL as individual commits via session.execute() + session.commit() per statement. Then alembic stamp to sync state. Alternatively: paste SQL in Supabase SQL Editor.
Consequences
Positive: Migrations work reliably. Alembic history is accurate. Negative: Manual two-step process. Easy to forget the stamp. New engineers will hit this silently.
